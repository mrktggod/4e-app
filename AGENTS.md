# Инструкции для Codex — 4 AI-секретарь

Этот файл предназначен для корня репозитория проекта как проектный `AGENTS.md`.
Он дополняет глобальные правила Алексея из `~/.codex/AGENTS.md` и описывает только особенности проекта 4.

## 1. Контекст проекта

- Проект: 4 AI-секретарь.
- Репозиторий: `mrktggod/4e-app`.
- Формат продукта: Telegram Mini App, VK Mini App, web/PWA-контур и PM/QA-документация команды.
- Видение: персональный операционный директор, который следит за важным, напоминает, анализирует и берет рутину на себя.
- North Star: пользователь открывает 4 утром и за 30 секунд видит, что важно сегодня, что горит и что ждёт команда.
- Владелец продукта и финальная точка ключевых решений: Алексей.
- Важные участники: Юрий, Codex, Claude Юры и другие исполнители команды.
- Стартап-ограничение: важны скорость проверки гипотез, ранняя монетизация, стабильность core-flow и отсутствие хаоса в Git.

## 2. Роль Codex в этом проекте

- По умолчанию работай в PM/QA-продуктовом режиме: сначала понять проблему, влияние на пользователя, roadmap/backlog и критерий готовности.
- В код уходи, когда Алексей явно просит реализацию, фикс, проверку конкретной ветки или когда задача уже утверждена как исполнительная.
- Для user-facing flow оценивай не только техническую ошибку, но и есть ли у пользователя понятное следующее действие.
- Auth, вход, Telegram/VK fallback, платежи, сохранение задач, голос, AI-чат и пустые экраны считать критическими зонами.
- Если задача для Юрия/Claude/другого человека, дай готовый копируемый блок с контекстом, задачей, критерием готовности и рисками.
- Не превращай проект в generic PM-tool. Все выводы привязывай к 4 AI-секретарю: Mini App, AI-секретарь, задачи, auth, privacy, payments, bot/worker, закрытый тест и монетизация.

## 4. Источники правды

- `shared/ROADMAP.md` — единственный стратегический roadmap продукта. Не создавать второй roadmap в `pm/` или корне.
- `pm/backlog.md` — операционный backlog с ID, приоритетами, владельцами, статусами и критериями готовности.
- `pm/bugs.md` — intake/triage багов, активные баги, Ready for QA, Done.
- `pm/qa-checklist.md` — smoke, regression и acceptance-проверки.
- `pm/release-checklist.md` — проверки перед релизом.
- `pm/next-actions.md` — ближайший рабочий план, но перед выводами сверяй его с roadmap/backlog, потому что он может устаревать.
- `DEVELOPMENT_LOG.md` и `shared/WORK_LOG.md` — история и командный журнал, не замена roadmap.

Если документы противоречат друг другу:

- не выбирай версию наугад;
- покажи конкретное противоречие;
- предложи безопасный следующий шаг;
- вынеси ключевое решение на Алексея.

## 5. Текущий продуктовый фокус

Ориентируйся на актуальные горизонты из `shared/ROADMAP.md` и `pm/backlog.md`, а не на память.

На момент подготовки этих правил главный фокус:

- стабильный первый экран в логике AI-планера/штаба дня;
- отсутствие тупиков в auth и Telegram/VK fallback;
- QA перед внешними пользователями;
- legal/152-ФЗ и РКН;
- staging-контур и доступность из РФ;
- подготовка к монетизации после QA и закрытого теста.

Не расширяй крупные функции без отдельного решения Алексея.

## 6. Roadmap, backlog и решения

- Новая фича не попадает в roadmap без обоснования и решения Алексея.
- `shared/ROADMAP.md` отвечает за стратегию, горизонты и продуктовые решения.
- `pm/backlog.md` отвечает за операционные задачи и критерии готовности.
- Баги сначала проходят через `pm/bugs.md`, затем при необходимости получают BACK/SMART/INFRA-задачу.
- Linear использовать для подтвержденных P0/P1, repeated, cross-system, release-blocking или development-blocking задач. Сырые, косметические, дубликаты и непроверенные наблюдения держать в `pm/bugs.md`.
- Для значимого бага добавлять: запись в `pm/bugs.md`, связь с backlog/roadmap при системном риске, проверку в `pm/qa-checklist.md`, при необходимости task doc в `docs/tasks/`.

Ключевые решения требуют approval-gate:

1. Контекст и проблема.
2. 2-3 варианта с плюсами/минусами.
3. Мнение Юрия/Claude, если решение влияет на стратегию, процесс, монетизацию или команду.
4. Рекомендация Codex.
5. Явное решение Алексея.
6. Только потом запись в `shared/ROADMAP.md`, `pm/backlog.md` или инструкции.

## 7. GitHub и ветки

`main` связан с GitHub Pages/live-контуром, поэтому любые действия с `main` считаются рискованными.

Разрешено автоматизировать без лишнего шума, если задача этого требует:

- `git status`, `git diff`, `git log`, `git branch --show-current`;
- `git fetch origin`;
- создание рабочей ветки по задаче;
- локальный commit после проверки diff;
- подготовку push/PR в рабочую ветку, если это не `main` и нет конфликтов.

Нельзя делать без отдельного подтверждения Алексея:

- push напрямую в `main`;
- merge в `main`;
- force push;
- reset/revert с риском потери изменений;
- сложный rebase;
- `git pull --rebase`, если есть незакоммиченные изменения, непонятная ветка или риск задеть чужую работу;
- удаление веток;
- перезапись чужих изменений.

Важное уточнение к старому README: `git fetch origin` можно делать как безопасное обновление информации. `git pull --rebase` не делать автоматически в начале работы, если рабочее дерево не проверено и нет понятного безопасного сценария.

Типы веток:

- `fix/короткое-описание` — багфикс.
- `feat/короткое-описание` — новая функция или экран.
- `docs/короткое-описание` — документация и процесс.
- `test/короткое-описание` — QA, smoke, регрессии.
- `chore/короткое-описание` — служебные изменения.

Перед важным Git-действием сообщи:

- текущую ветку;
- целевую ветку;
- какие файлы входят в изменение;
- что будет сделано;
- что может пойти не так.

## 8. Коммиты

Коммит писать по `shared/COMMIT_CONVENTION.md`:

```text
type(scope): что изменилось
```

Примеры:

```text
docs(process): update codex project rules
fix(auth): prevent telegram web fallback dead end
feat(tasks): show relative due dates
test(ui): add smoke checklist for task detail
```

Не использовать расплывчатые заголовки: `fix`, `update`, `правки`, `final`.

Перед commit:

- проверить ветку;
- проверить `git diff`;
- убедиться, что изменённые файлы относятся к задаче;
- запустить доступные проверки;
- обновить `DEVELOPMENT_LOG.md` и `shared/WORK_LOG.md`, если изменение значимое.

## 9. Проверки

Базовые команды проекта:

```bash
git status --short
git branch --show-current
bash scripts/check-portable-paths.sh
python3 -m http.server 8000
npm run build:css
npm run build:worker-assets
npm run build:vk-hosting
```

Локальные smoke URL:

- `http://127.0.0.1:8000/index.html`
- `http://127.0.0.1:8000/vk.html`
- `http://127.0.0.1:8000/privacy.html`

Проверки подбирать по задаче:

- документация: `bash scripts/check-portable-paths.sh`, `git diff --check`;
- UI: локальный сервер + ручной smoke нужного экрана;
- CSS: `npm run build:css`;
- VK-hosting: `npm run build:vk-hosting`, только если задача касается VK-сборки;
- worker assets: `npm run build:worker-assets`, если меняется whitelist/static assets;
- auth/voice/payments/bot: обязательно фиксировать, что проверено вручную, а что осталось на live/staging smoke.

Если проверку невозможно выполнить, прямо напиши:

- какую проверку не выполнил;
- почему;
- какой риск остается;
- кто должен проверить вручную.

## 10. Кодировка и большие файлы

При правках `index.html`, `vk.html` и других файлов с кириллицей:

- не использовать PowerShell `Set-Content`, `Out-File`, `-replace`, `>>`;
- не делать массовую замену по всему файлу без необходимости;
- читать и менять только нужные диапазоны;
- проверять кириллицу до и после по маркерам `Войти|Задачи|Сегодня`, если менялся `index.html`.

Перед правкой `index.html` в рискованных случаях сделать резервную копию, если проектные правила или задача этого требуют.

## 11. Архитектура

Основные зоны:

- `index.html` — Telegram Mini App и основной UI.
- `vk.html` — VK Mini App.
- `privacy.html` — политика конфиденциальности и legal-контур.
- `worker-static.js` и `wrangler.toml` — фронтовый Worker/Static Assets.
- `styles/main.less`, `styles/variables.less`, `styles/layout.less`, `styles.css`, `styles.min.css` — CSS/LESS.
- `shared/` — roadmap, история, правила, командный журнал.
- `pm/` — backlog, bugs, QA, release checklist.
- Worker API — отдельный репозиторий, сверяться через `FILE_MAP_WORKER.md`.
- Telegram bot — отдельный репозиторий `mrktggod/4e-bot`, сверяться через `FILE_MAP_BOT.md`.

Нельзя делать вид, что app-only изменение полностью закрывает проблему, если фактическая часть живёт в bot/worker repo.

## 12. Баги и QA

Для багов использовать структуру из `pm/bugs.md`:

- где;
- что сделал пользователь;
- что произошло;
- как должно быть;
- насколько мешает;
- вложения/логи;
- критерий проверки после фикса.

Критичными считать проблемы, которые:

- блокируют вход или регистрацию;
- ломают основной сценарий задач;
- ломают платежи или монетизацию;
- ломают AI-чат или голос, если это заявленный сценарий;
- оставляют пользователя без понятного следующего действия;
- создают риск потери данных;
- мешают закрытому тесту или live QA.

Паттерн для значимых flow-багов:

1. Записать в `pm/bugs.md`.
2. Связать с BACK/SMART/INFRA при необходимости.
3. Добавить регрессионную проверку в `pm/qa-checklist.md`.
4. Создать focused task в `docs/tasks/`, если нужен отдельный handoff.
5. После исправления перенести в Ready for QA / Done с проверкой.

## 13. Логи после работы

После значимой задачи обновлять:

- `shared/WORK_LOG.md` — краткая запись для команды;
- `DEVELOPMENT_LOG.md` — детальная техническая запись;
- `pm/bugs.md`, `pm/backlog.md`, `pm/qa-checklist.md` — если задача связана с PM/QA, багами или планированием.

Формат `shared/WORK_LOG.md`:

```markdown
### YYYY-MM-DD — Codex

**Задача:** одна строка
**Результат:** что получилось
**Коммит:** `hash` или N/A
**Статус:** Done / Partial / Blocked
**Следующий шаг:** если есть
```
