# Правила работы с файлами проекта

Кто пишет куда, когда и зачем. Источник истины для всей команды.

---

## Файловая структура

| Файл | Кто владелец | Что хранится | Кто пишет |
|---|---|---|---|
| `shared/ROADMAP.md` | Юрий | Стратегия, горизонты, столпы, решения | Юрий + Мимо (Cowork) |
| `pm/backlog.md` | Алексей | Все задачи с ID, приоритетом, статусом | Алексей + Мимо + Codex |
| `pm/bugs.md` | Алексей | Баги из тестирования | Алексей + тестировщики |
| `shared/WORK_LOG.md` | Команда | Краткий журнал завершённых задач | Все агенты + Мимо |
| `DEVELOPMENT_LOG.md` | Codex | Детальный техлог изменений | Codex |
| `AGENTS.md` | Алексей | Инструкции для Codex | Алексей + Мимо |
| `docs/tasks/` | Юрий / Алексей | Атомарные задачи для Кодекса | Юрий + Мимо |
| `FILE_MAP.md` / `FILE_MAP_*.md` | Codex | Карта файлов проекта | Codex (обновляет после правок) |

## Правила по каждому файлу

### shared/ROADMAP.md — стратегия

- **Не писать** BACK-xxx задачи — только горизонты и направления
- **Писать:** когда меняется стратегия, позиционирование, добавляется горизонт
- **Кто решает:** Юрий (через Мимо / Cowork)
- **Синхронизировать** с `pm/backlog.md` при смене горизонта

### pm/backlog.md — задачи

- **Каждая задача** должна иметь: ID, тип, приоритет, ответственный, статус, критерий готовности
- **Статусы:** `Todo` → `In Progress` → `Ready for QA` → `Done` (или `Known issue` / `Manual blocker`)
- **Кто обновляет статус:** тот кто завершил задачу (Codex — в конце сессии, Алексей — после ручных действий, Мимо — после UI/docs правок)
- **Не создавать задачи без ID** — следующий свободный номер смотреть в конце таблицы
- **BACK-018 и выше** — текущий горизонт

### shared/WORK_LOG.md — журнал

- **Писать после каждой** завершённой задачи — одна запись = одна задача
- **Формат строго:** дата, агент, задача, результат, коммит, статус, следующий шаг
- **Добавлять сверху** в секцию `## Лог` — свежие записи первыми
- **Не дублировать** детали из `DEVELOPMENT_LOG.md`

### pm/bugs.md — баги

- **Писать:** только подтверждённые баги из тестирования
- **Формат:** шаги воспроизведения, ожидаемое vs реальное, приоритет
- **Не писать** в bugs.md то что идёт в backlog — баги P0/P1 создаются отдельными BACK-xxx
- **Колонка BACK-xxx** — заполнять при создании задачи в backlog по итогам бага; `—` если задачи нет

### docs/tasks/ — задачи для Кодекса

- **Создавать** для каждой нетривиальной задачи Кодекса (>1 файл или >30 строк)
- **Формат:** что делать, файлы для чтения, критерий готовности
- **После выполнения** Кодекс оставляет файл — не удалять, он часть истории

## Правила Git

| Ситуация | Правило |
|---|---|
| Любая правка index.html или worker.js | Новая ветка `fix/*` или `feat/*` |
| Документация / backlog / логи | Можно прямо в `main` или ветка `docs/*` |
| Hotfix P0 | Прямо в `main` с записью в WORK_LOG |
| Мерж чужой ветки | Сначала `git fetch`, потом `--no-edit` или PR |
| Конфликты в лог-файлах | `--theirs` (берём чужую версию, не теряем свою через WORK_LOG-запись) |

### Кто мержит

- **Кодекс** — создаёт PR, мержит сам через CLI если нет конфликтов
- **Алексей** — мержит через GitHub Desktop или CLI, проверяет перед мержем
- **Юрий** — мержит через Мимо (Cowork) командой в терминале

## Связь между файлами

| Откуда | Куда | Как |
|---|---|---|
| `shared/ROADMAP.md` | `pm/backlog.md` | Колонка **Горизонт** (Г1 / Г2 / Г3) в каждой задаче |
| `pm/backlog.md` | `pm/bugs.md` | Колонка **BACK-xxx** в таблицах активных/закрытых багов |
| `pm/bugs.md` | `pm/backlog.md` | Ссылка BACK-xxx в деталях бага — когда баг превращается в задачу |

**Правило:** если создаёшь баг и сразу создаёшь BACK-задачу — проставь обе ссылки в тот же коммит.

## Синхронизация с GitHub

Все PM-файлы (`pm/backlog.md`, `pm/bugs.md`, `shared/ROADMAP.md`) живут в общем git-репозитории.

- **Алексей** пушит изменения в `main` после каждого спринта или значимого обновления
- **Юрий** перед работой делает `git pull` чтобы получить свежую версию файлов
- **Мимо (Cowork)** читает файлы из локальной папки — поэтому важно сначала `git pull`, потом работать
- **Конфликты** в PM-файлах: разбирать вручную, не использовать `--theirs` (в логах важна каждая строчка)

## Что НЕ делать

- ❌ Писать стратегию в backlog — это в ROADMAP
- ❌ Писать конкретные задачи в ROADMAP — это в backlog
- ❌ Создавать задачи без ID
- ❌ Редактировать `worker.js` напрямую — только через `src/*.js` → `npm run build`
- ❌ Коммитить в main без проверки кириллицы в `index.html`
- ❌ Забывать обновлять статус задачи в backlog после выполнения
