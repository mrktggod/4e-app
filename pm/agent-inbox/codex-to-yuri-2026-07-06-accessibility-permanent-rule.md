## ЗАДАЧА ДЛЯ ЮРЫ

```text
ЗАДАЧА ДЛЯ ЮРЫ

Контекст:
Accessibility теперь постоянное правило UI-разработки 4, а не разовая полировка BACK-050. Если ты меняешь экран, форму, навигацию, модалку, bottom sheet, toast/status или интерактивную карточку, нужно сохранять базовую доступность этого сценария.

Что нужно применять всегда:
1. Поля ввода имеют label, aria-label или другой понятный accessible name. Placeholder не считается полноценной подписью.
2. Ошибки форм связаны с полями через aria-describedby/aria-invalid или эквивалент.
3. Интерактивные элементы доступны с клавиатуры, имеют видимый focus и логичный Tab / Shift+Tab порядок.
4. Обычные сообщения доступны как role="status" / aria-live, критические ошибки — как role="alert" или полевые ошибки.
5. Dialog / bottom sheet / consent-экраны получают фокус внутрь, имеют понятный cancel/close и возвращают фокус после закрытия.
6. Кнопки, nav, tabs, иконки и быстрые действия имеют достаточную область нажатия на mobile.
7. Декоративные SVG/анимации не засоряют accessibility tree; при заметной анимации учитывать prefers-reduced-motion.

Как работать со старым кодом:
Не переписывать весь index.html ради accessibility одним проходом. Но если трогаешь конкретный экран, не добавляй новый недоступный UI и по возможности доводи этот сценарий до baseline.

Проверки:
- FILE_MAP_UI.md перед чтением index.html/vk.html.
- Проверка кириллицы до/после, если менялся index.html.
- npm run build:css, если менялись LESS.
- bash scripts/check-ui-architecture.sh.
- bash scripts/check-portable-paths.sh.
- git diff --check.
- Ручной keyboard smoke: Tab, Shift+Tab, Enter, Escape.
- Mobile smoke: 360/375/390px.

Критерий готовности:
Новый или изменённый UI не увеличивает inline-долг и не добавляет accessibility-долг. Пользователь может пройти ключевой сценарий с понятным фокусом, подписями, ошибками, статусами и нормальными зонами нажатия.

Куда вернуть результат:
Ветка задачи, запись в shared/WORK_LOG.md и DEVELOPMENT_LOG.md, кратко указать какие accessibility-проверки пройдены.
```
