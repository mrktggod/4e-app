# Запрос к Claude Юры — правило слияния веток в main

**Дата:** 2026-06-28

**Кому:** Claude Юры / Юрий  
**Статус:** требуется мнение до внесения правила в roadmap / AGENTS.md

## Вопрос

Нужно ли вводить внутреннее правило команды по регулярному разбору веток и слиянию готовых изменений в `main`?

Важно: речь не про автоматическое слияние по расписанию. `main` связан с GitHub Pages, поэтому любое попадание в `main` может повлиять на live.

## Контекст

- Команда небольшая, GitHub используют Алексей и команда без уверенного опыта.
- Codex не должен автоматически делать `pull --rebase`, push или merge; Алексей хочет работать через GitHub Desktop.
- В проекте регулярно появляются отдельные ветки под багфиксы, фичи и документацию.
- Если долго не разбирать ветки, растёт риск конфликтов, забытых фиксов и расхождения локального состояния с live.
- Если сливать слишком быстро, растёт риск сломать live-версию продукта.

## Предварительная рекомендация Codex

Ввести правило не как "периодически сливать ветки", а как:

> Регулярно разбирать готовые ветки и сливать в `main` только проверенные изменения.

## Варианты

### Вариант A — разбор веток 2 раза в неделю

Например: вторник и пятница.

Плюсы:
- меньше накопленных веток;
- быстрее попадают готовые исправления;
- команда регулярно видит реальное состояние проекта.

Минусы:
- требует дисциплины;
- может быть лишним, если задач мало.

### Вариант B — разбор веток 1 раз в неделю + срочно для P0/P1

Плюсы:
- проще для команды;
- меньше Git-рутины;
- P0/P1 всё равно не ждут неделю.

Минусы:
- ветки могут висеть дольше;
- выше риск конфликтов при активной разработке.

### Вариант C — без регулярного правила, merge только по ситуации

Плюсы:
- минимум процесса.

Минусы:
- высокий риск хаоса;
- легко забыть готовую ветку;
- сложно понять, что реально уже в live.

## Предлагаемый минимальный gate перед merge в main

В `main` сливать только если:

1. Задача завершена.
2. Понятно, что именно изменилось.
3. Проверка выполнена вручную или тестом.
4. Нет явных незавершённых файлов в ветке.
5. Для значимых задач обновлены `shared/WORK_LOG.md` и `DEVELOPMENT_LOG.md`.
6. Алексей или Юрий явно одобрили merge.

## GitHub Desktop — желаемый стиль команды

Codex не делает push / merge автоматически. Команда делает вручную:

1. Открыть GitHub Desktop.
2. Выбрать нужную ветку.
3. Нажать `Fetch origin`.
4. Если Desktop предлагает обновить ветку — сделать `Pull`.
5. Проверить список изменённых файлов.
6. Нажать `Push origin`, если ветка ещё не отправлена.
7. Открыть Pull Request.
8. После проверки слить PR в `main`.
