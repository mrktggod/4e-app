# BACK-050 — Accessibility baseline критических сценариев

**Приоритет:** P1
**Ответственный:** Юрий + Codex
**Рекомендуемые ветки:**
- `fix/accessibility-auth-baseline`
- `fix/accessibility-status-dialogs`
- `fix/accessibility-focus-targets`

## Контекст

После изучения курса HTML Academy «Доступность веб-интерфейсов» фиксируем доступность как часть качества продукта перед закрытым тестом. Для 4 это не отдельная декоративная полировка, а снижение риска в критических сценариях: вход, регистрация, сброс пароля, создание задачи, голосовое согласие, навигация и системные сообщения.

Ориентир по смыслу: WCAG 2.2 / практический уровень для Mini App, без попытки сразу заявлять полное соответствие стандарту.

## Цель

Пользователь может пройти основные сценарии без мыши, с понятным фокусом, читаемыми ошибками, доступными статусными сообщениями и нормальными зонами нажатия на мобильном экране.

## Порядок работы

### Шаг 1 — auth/forms baseline

**Ветка:** `fix/accessibility-auth-baseline`

Что сделать:

1. Пройти экран входа, регистрации, сброса пароля и нового пароля.
2. Добавить явные `label` или доступные подписи для полей, не полагаясь только на `placeholder`.
3. Связать ошибки с полями через `aria-describedby`/`aria-invalid` или эквивалентный доступный механизм.
4. Проверить keyboard flow: Tab, Shift+Tab, Enter, видимый фокус.
5. Не менять бизнес-логику auth, если задача решается разметкой/JS-событиями.

Где смотреть:

- `FILE_MAP_UI.md` — сначала открыть карту.
- `index.html` — экраны `forgot-password`, `reset-password`, `login`.
- `vk.html` — auth-экран VK Mini App, если берём в эту же ветку.
- `styles/**/*.less` — новые стили только через LESS.

Критерий готовности:

- Email/password login работает мышью и Enter.
- Пустые/невалидные поля показывают текстовую ошибку, связанную с конкретным полем.
- Табуляция не теряется и не прыгает в невидимые элементы.
- Фокус видим на кнопках, ссылках и полях.
- `git diff --check`, `bash scripts/check-portable-paths.sh`, `bash scripts/check-ui-architecture.sh` проходят.

### Шаг 2 — status/toast baseline

**Ветка:** `fix/accessibility-status-dialogs`

Что сделать:

1. Привести `#toast` к доступному статусному сообщению: `role="status"`/`aria-live="polite"` для обычных уведомлений.
2. Для ошибок формы и критических ошибок использовать `role="alert"` или отдельные полевые ошибки, а не только временный toast.
3. Проверить сценарии: пустой login, невалидный forgot password, ошибка сети, успешная отправка письма, успешное сохранение задачи.
4. Не оставлять важный следующий шаг только во всплывающем toast.

Критерий готовности:

- Системные сообщения остаются понятными без чтения исчезающего toast.
- Ошибки форм видны рядом с полями или в понятном блоке.
- Success/error статусы не ломают текущий визуал.
- Нет роста inline UI-долга.

### Шаг 3 — dialogs/focus/touch-targets

**Ветка:** `fix/accessibility-focus-targets`

Что сделать:

1. Проверить quick-add overlay, contact panel и biometric consent как dialog/bottom sheet.
2. Добавить `role="dialog"`, `aria-modal`, связку с заголовком и понятный cancel/close, где это безопасно.
3. При открытии переносить фокус внутрь, при закрытии возвращать фокус в исходную кнопку.
4. Проверить Escape/back/cancel, чтобы пользователь не застревал.
5. Проверить зоны нажатия нижней навигации, tab-кнопок, маленьких иконок и быстрых действий на 360/375/390px.
6. Для декоративных SVG/анимаций добавить `aria-hidden="true"`, где они не несут смысла.
7. Учесть `prefers-reduced-motion` для декоративной анимации на Home, если правка не раздувает задачу.

Критерий готовности:

- Bottom sheets не ловят пользователя в тупик.
- Фокус не уходит за модалку и не теряется после закрытия.
- Ключевые действия нажимаются пальцем без точного попадания в мелкий текст.
- Декоративные элементы не засоряют accessibility tree.

## Что не делать

- Не заявлять полное WCAG-соответствие продукта.
- Не переписывать весь `index.html` одним проходом.
- Не менять позиционирование продукта или визуальный стиль без отдельной задачи.
- Не добавлять новые inline `style`/`onclick`/`oninput` в рамках новых правок.
- Не ломать Telegram/VK Mini App ради desktop-only доступности.

## Проверка после каждого шага

1. Если менялся `index.html`, выполнить проверку кириллицы до/после по правилу проекта.
2. `npm run build:css`, если менялись LESS-файлы.
3. `bash scripts/check-ui-architecture.sh`.
4. `bash scripts/check-portable-paths.sh`.
5. `git diff --check`.
6. Ручной smoke на 360/375/390px.
7. Ручной keyboard smoke: Tab, Shift+Tab, Enter, Escape.
