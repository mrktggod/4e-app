# BACK-058 — профиль из OAuth: email / phone / birthdate

Статус документа: product/legal decision, 2026-07-16.

## Решение

Не запрашивать персональные поля "про запас".

Профильные данные из OAuth нужны только там, где они дают понятную пользу пользователю:

- email — восстановление доступа, чеки, аккаунтная связка;
- phone — доверие/контакт, если пользователь сам разрешил;
- birthdate — только если появится конкретный сценарий персонализации, а не для анкеты.

## Privacy principle

Каждое поле — отдельное объяснение и отдельное согласие.

Текст должен отвечать:

- зачем 4 просит поле;
- где поле будет видно;
- можно ли удалить/отозвать;
- что будет, если не дать поле.

## MVP order

1. Не трогать до закрытого beta-run и стабилизации `BACK-045`.
2. Сначала email как account trust field.
3. Потом phone только при явной необходимости.
4. Birthdate — later, если появится доказанная value.

## UI rule

В профиле показывать source/status:

```text
Email: подтверждён через VK ID
Телефон: не подключён
Дата рождения: не запрашивается
```

Не показывать пустые поля так, будто пользователь "не заполнил анкету". Лучше:

```text
Не подключено
```

## OAuth provider boundaries

Telegram:

- не считать источником email/phone/birthdate;
- контакт через bot button — отдельное действие пользователя.

VK ID:

- email/phone только если scope и consent явно выданы;
- не использовать как источник birthdate, если провайдер не даёт его надёжно.

Яндекс ID:

- использовать только после готовых client id/secret и отдельного consent smoke;
- birthdate не просить до продуктового сценария.

## Product trigger для birthdate

Birthdate можно обсуждать только если есть один из сценариев:

- персональный годовой/недельный ритуал;
- legal/age requirement;
- явно полезная персонализация, подтверждённая beta feedback.

Если сценария нет — не просим.

## Definition of Done

`BACK-058` можно брать в реализацию, когда:

- `BACK-045` полностью прошёл live smoke;
- выбран первый provider/field;
- consent copy утверждён;
- есть экран удаления/отзыва или понятный путь удаления;
- QA проверяет, что отказ от поля не ломает signup/login.
