# SMART-013 — AI-декомпозиция задачи на этапы

**Дата:** 2026-07-08
**Владелец:** Codex / Юрий
**Тип:** Product/AI/UI
**Приоритет:** P2
**Статус:** Triaged
**Рекомендуемая ветка:** `feat/smart-013-task-decomposition`

## Контекст

4 позиционируется не как обычный список задач, а как AI-планер / персональный операционный директор. Поэтому сильный сценарий карточки задачи — не просто хранить текст, срок и статус, а помогать пользователю понять следующий практический шаг.

В `BACK-022` уже добавлен чек-лист внутри задачи. В `BACK-044` карточку специально упростили, чтобы первый экран не был перегружен. Поэтому SMART-013 нельзя делать как новую тяжёлую сущность "подзадачи". Правильный MVP: **AI предлагает этапы, а пользователь подтверждает их как обычные пункты чек-листа**.

Продуктовый смысл:

```text
Пользователь записал задачу -> 4 превратил её в короткий план -> пользователь начал действовать.
```

## Решение

Добавить в блок `Чек-лист` на экране `task-detail` кнопку:

```text
Разбить на этапы
```

После нажатия AI анализирует название, описание, срок, приоритет и текущий чек-лист задачи. Затем показывает превью 3-7 этапов. Пользователь должен явно подтвердить добавление.

Главное правило: **AI ничего не записывает в задачу без подтверждения пользователя**.

Это не финальный дизайн и не pixel-perfect макет. Его задача — показать продуктовую логику: CTA живёт внутри чек-листа, AI открывает preview, пользователь подтверждает запись этапов.

## UX-сценарий

### 1. Чек-лист пустой

В блоке чек-листа показывать нейтральное пустое состояние:

```text
Пока нет этапов
4 может разложить задачу на понятные шаги.

[Разбить на этапы]
```

### 2. Чек-лист уже есть

Кнопка остаётся вторичным действием:

```text
[Добавить этапы с AI]
```

Если пользователь подтверждает генерацию при непустом чек-листе, новые этапы добавляются в конец. Существующие пункты не удалять и не перезаписывать.

### 3. AI формирует превью

Показать bottom sheet или компактный preview-блок:

```text
4 предлагает 5 этапов

1. Уточнить цель и результат
2. Собрать вводные
3. Подготовить первый вариант
4. Проверить с человеком
5. Зафиксировать итог

[Добавить в чек-лист] [Сгенерировать заново] [Отмена]
```

### 4. Пользователь подтверждает

После нажатия `Добавить в чек-лист` этапы становятся обычными пунктами `checklist`:

```js
{ text: "Уточнить цель и результат", done: false }
```

Дальше они редактируются, отмечаются и сохраняются так же, как ручной чек-лист.

## Поведение AI

AI должен возвращать короткие практические этапы, а не длинный совет.

Правила генерации:

1. 3-7 пунктов.
2. Каждый пункт до 70 символов.
3. Пункты начинаются с действия: `Уточнить`, `Собрать`, `Написать`, `Проверить`, `Отправить`.
4. Не добавлять общие пустые шаги вроде `Сделать задачу`.
5. Не выдумывать людей, даты, документы и внешние факты, если их нет в задаче.
6. Если задача уже простая, вернуть 2-3 шага или показать текст: `Задача уже простая, можно выполнить без декомпозиции`.
7. Если в задаче есть срок, учитывать его в логике порядка, но не создавать отдельные дедлайны на каждый этап в MVP.

Желательный формат ответа AI:

```json
{
  "steps": [
    {"text": "Уточнить цель и результат"},
    {"text": "Собрать вводные"},
    {"text": "Подготовить первый вариант"}
  ],
  "note": "Короткая причина, почему такие этапы подходят"
}
```

Если модель вернула невалидный JSON, приложение должно показать мягкую ошибку и не менять задачу.

## Стартовые шаблоны декомпозиции

AI может ориентироваться на тип задачи:

| Тип задачи | Пример этапов |
| --- | --- |
| Документ / отчёт | собрать вводные, сделать черновик, проверить факты, отправить |
| Запуск / проект | определить результат, разбить на блоки, сделать первый шаг, проверить риски |
| Встреча / звонок | подготовить вопросы, провести разговор, зафиксировать итог |
| Событие | список гостей/участников, место/время, материалы, напоминание |
| Покупка / бытовая задача | уточнить критерии, выбрать вариант, проверить цену/срок, выполнить |
| Ремонт / переезд | список работ, материалы, исполнитель, контроль результата |

Это не жёсткие кнопки для пользователя. Это ориентиры для prompt-логики.

## Что входит в MVP

1. Кнопка `Разбить на этапы` в блоке чек-листа на `task-detail`.
2. Loading-состояние: `4 разбивает задачу на этапы...`.
3. AI-запрос по текущей задаче.
4. Preview 3-7 шагов до записи в задачу.
5. Действия: `Добавить в чек-лист`, `Сгенерировать заново`, `Отмена`.
6. Запись подтверждённых этапов в существующее поле `checklist`.
7. Сохранение через текущий механизм `saveTaskEdits()`.
8. Мягкие ошибки при сбое AI или невалидном ответе.
9. Сохранение ручного редактирования чек-листа после генерации.

## Что не входит в MVP

- Вложенные подзадачи.
- Отдельная вкладка `Этапы`.
- Проценты выполнения задачи.
- Дедлайн, статус, приоритет или исполнитель на каждый этап.
- Зависимости между этапами.
- Автоматическая генерация этапов для всех задач.
- Автоматическая запись без подтверждения.
- Уведомления по каждому этапу.
- Отдельная таблица в D1 под этапы.
- Командное согласование этапов.
- Углубление в каждый шаг через чат задачи.

Углубление в шаги относится к будущей связке с `BACK-032` и должно проектироваться отдельно.

## Технические ориентиры

Основная зона:

- `index.html` — экран `task-detail`, блок чек-листа, функции открытия задачи и сохранения.
- `styles/screens/tasks.less` — стили чек-листа, preview/bottom sheet и mobile-состояния.
- `styles.css` / `styles.min.css` — обновить после LESS-сборки, если проект сейчас требует синхронизации CSS.
- `FILE_MAP_UI.md` — обновить, если добавятся новые функции или заметно изменятся диапазоны.

Использовать существующую модель чек-листа:

```js
checklist: [
  { text: "Первый этап", done: false }
]
```

Не вводить новую модель данных без отдельного решения.

## Acceptance criteria

1. В пустом чек-листе есть понятный CTA `Разбить на этапы`.
2. При нажатии появляется loading-состояние, интерфейс не зависает.
3. AI предлагает 3-7 коротких этапов.
4. Этапы не записываются в задачу до подтверждения пользователя.
5. После подтверждения этапы появляются как обычные пункты чек-листа.
6. Существующий чек-лист не перезаписывается.
7. Пользователь может отметить, удалить и отредактировать AI-этапы вручную.
8. После reload этапы остаются в задаче.
9. При ошибке AI задача не меняется, пользователь видит понятное сообщение.
10. На мобильной ширине preview и кнопки не налезают на нижнюю панель.

## QA checklist

Проверить на mobile viewport и в Telegram WebView:

1. Задача без чек-листа -> `Разбить на этапы` -> preview -> `Добавить в чек-лист`.
2. Задача с существующим чек-листом -> генерация добавляет пункты в конец.
3. `Отмена` закрывает preview и ничего не сохраняет.
4. `Сгенерировать заново` заменяет только preview, но не текущий чек-лист.
5. Ошибка AI / пустой ответ / невалидный JSON не ломает карточку.
6. После отметки пунктов и reload состояние сохраняется.
7. Старая задача без поля `checklist` открывается без пустого экрана.

## Риски

| Риск | Как снизить |
| --- | --- |
| Карточка снова станет перегруженной | Не делать отдельную вкладку и не показывать генерацию выше срока/статуса |
| AI будет писать слишком общие шаги | Ограничить prompt: короткие action-пункты, без общих советов |
| Пользователь не доверяет автозаписи | Только preview и явное подтверждение |
| Раздувание модели данных | Использовать существующий `checklist`, не вводить `subtasks` |
| Ошибки AI испортят задачу | При ошибке не менять локальный cache и не вызывать save |
