# BACK-066 - VK stable line functional parity

**Дата:** 2026-07-21
**Владелец:** Codex / Yuri / Alexey QA
**Тип:** Product/Platform/UX
**Приоритет:** P1
**Статус:** Triaged

## Контекст

Алексей проверил VK-версию как браузерное приложение на телефоне: вход работает, приложение открывается и субъективно ведет себя стабильнее остальных поверхностей. Это делает VK важной стабильной линией продукта, а не только вспомогательным Mini App-адаптером.

При этом VK-версия недобита по функционалу: часть информации и сценариев обрезаны относительно основной версии. Сейчас в roadmap уже закрыта инфраструктура VK-доступа (`INFRA-004`, `INFRA-005`), но нет отдельной продуктовой задачи на доведение VK-поверхности до полноценного пользовательского контура.

## Проблема

`vk.html` живет отдельной поверхностью и исторически содержит урезанный набор экранов/логики:

- auth/login работает, но после входа нужно проверить полноту home/tasks/AI/calendar/stats/profile;
- часть данных может не отображаться или отображаться короче, чем в основной версии;
- UX может быть стабильным за счет простоты, но функционально отставать от `index.html`;
- без parity-аудита команда рискует чинить VK точечно, не понимая, какие сценарии реально должны стать основной стабильной линией.

## Цель

Сделать VK-версию осознанной стабильной линией:

1. Зафиксировать минимальный набор сценариев, который обязан работать в VK.
2. Сравнить `vk.html` с основной поверхностью по данным, экранам и действиям.
3. Добить недостающую информацию и обрезанные элементы без большого архитектурного распила.
4. Подтвердить стабильность живым mobile/VK smoke.

## Scope

Включить в аудит и доведение:

- вход VK и возврат в приложение;
- главный экран / план дня;
- список задач и карточка задачи;
- создание задачи из текста/чата, где это уже поддержано;
- AI-чат;
- календарь;
- статистика;
- профиль и account state;
- пустые, loading и error-состояния;
- нижняя навигация, safe-area, мобильная читаемость.

## Acceptance criteria

- Составлена parity-таблица `index.html` vs `vk.html`: экран, данные, действия, статус `ok/missing/partial`.
- Для P1-сценариев VK не теряет ключевую информацию, которая есть в основной версии: название задачи, срок, статус, приоритет, человек, описание/обсуждение, AI-ответы, календарные дедлайны, базовая статистика.
- VK-версия не показывает пустые блоки там, где данные реально есть.
- VK-версия не ломает вход и session state после обновления страницы.
- На телефоне в VK/browser shell проходят smoke-сценарии: login -> home -> task list -> task detail -> create/edit task -> AI chat -> calendar -> profile.
- Для отсутствующих функций есть явные graceful states, а не ощущение обрезанного интерфейса.

## Не делать в этом task

- Не включать VK Pay и не менять payment/entitlement логику.
- Не делать production deploy без отдельного подтверждения.
- Не начинать большой `ARCH-001` распил монолита.
- Не переносить все из `index.html` механически, если для стабильной VK-линии достаточно узкого решения.

## Предлагаемый первый шаг

Сделать source/runtime-аудит VK:

```text
Экран | Есть в vk.html | Есть в index.html | Данные полные | Действия работают | Комментарий
```

После этого выделить 3-5 самых заметных функциональных дыр и закрывать их отдельными маленькими PR.
