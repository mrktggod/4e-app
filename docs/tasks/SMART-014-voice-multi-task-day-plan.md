# SMART-014 — Голосовой разбор нескольких задач и план дня

**Дата:** 2026-07-20
**Владелец:** Алексей / Юрий / Codex
**Тип:** Product / AI / Voice / UX / Worker
**Стратегический приоритет:** P1
**Очередность:** после закрытого beta-run и стабилизации `BACK-021`; не блокирует текущую beta
**Статус:** Triaged
**Горизонт:** Г2 / Later
**Рекомендуемая ветка реализации:** `feat/smart-014-voice-multi-task-plan`

## Коротко

Пользователь одним голосовым сообщением перечисляет несколько дел. 4 превращает поток речи в отдельные задачи, предлагает понятный порядок и смысловые группы, показывает превью и сохраняет задачи только после явного подтверждения.

```text
Пользователь выгрузил хаос из головы
→ 4 выделил отдельные дела
→ предложил главный фокус и порядок
→ пользователь подтвердил или изменил
→ задачи сохранились без выдуманных данных
```

Функция не является расширением `SMART-013`: `SMART-013` разбивает **одну задачу** на этапы чек-листа, а `SMART-014` выделяет **несколько самостоятельных задач** из одного входящего сообщения и собирает из них черновик плана.

## Проблема

Актуальный голосовой путь `sendVoiceMessage()` распознаёт task intent, вызывает `fallbackTaskFromText()` и передаёт результат в `createTaskFromChat()`. В итоге длинное перечисление может сохраниться как одна задача. Пользователь вынужден повторно разделять текст руками, а голос остаётся заменой клавиатуры вместо сильной функции личного AI-штаба.

Текущие связанные механики закрывают только части сценария:

- `SMART-008` умеет показывать превью действий и выполнять их после подтверждения;
- `SMART-009` задаёт один уточняющий вопрос для одной задачи;
- `SMART-010` ищет похожую существующую задачу;
- `SMART-013` декомпозирует одну задачу в чек-лист;
- `CAL-002` в будущем добавит длительность, временные слоты и оценку перегруза.

Нужен отдельный связующий сценарий, который не смешивает эти ответственности.

## Продуктовый смысл

4 должен продавать не хранение списка, а ощущение: «я выгрузил дела и понимаю, что делать дальше».

SMART-014 усиливает продукт сразу в четырёх местах:

1. **Быстрый захват:** несколько дел фиксируются одним сообщением.
2. **Фокус дня:** 4 предлагает решение, а не возвращает пользователю сырой список.
3. **Доверие:** предположения явно обозначаются, данные не записываются молча.
4. **Premium-ценность:** голос становится интеллектуальным планированием, а не диктовкой текста.

## Решение по приоритету

### Итог

**P1 стратегически, но не Now-блокер. Реализация — Г2 после beta gate.**

| Фактор | Оценка | Обоснование |
| --- | ---: | --- |
| Соответствие видению «личный штаб дня» | 5/5 | Даёт фокус и следующий шаг из реального входящего потока |
| Пользовательская ценность | 5/5 | Убирает ручное разделение и сортировку нескольких дел |
| Потенциал удержания | 4/5 | Может стать повторяемым ежедневным ритуалом |
| Premium / монетизация | 4/5 | Делает голос заметной платной ценностью, но требует проверки готовности платить |
| Срочность до beta | 2/5 | Не чинит текущий P0/P1 blocker и не должен задерживать beta-run |
| Сложность и риск | 3/5 | Нужны AI-контракт, новый preview-flow и надёжное пакетное сохранение |
| Уверенность | 4/5 | Сильное соответствие стратегии, но нет данных реального использования multi-task voice |

Почему не P2: функция относится к ядру позиционирования и прямо запрошена владельцем продукта.

Почему не P0: текущий продукт может пройти закрытую beta без неё.
Почему не начинать немедленно: сначала нужны стабильный голосовой smoke и фактический beta-сигнал, что пользователи создают реальные задачи и возвращаются.

### Gate старта реализации

Начинать runtime-разработку, когда выполнены условия:

1. `BACK-021` проходит живой голосовой smoke в Telegram Mini App на iOS и Android.
2. Проведён `BETA-001` либо Алексей отдельно решает запустить SMART-014 раньше.
3. У команды есть 10–20 обезличенных примеров реальных голосовых перечислений или подтверждение гипотезы минимум у 5 пользователей.
4. Утверждён MVP без календарного автопланирования.

## Jobs To Be Done

> Когда у меня в голове несколько разнородных дел, я хочу назвать их одной фразой, чтобы 4 сам превратил речь в понятный и контролируемый черновик плана, а я только проверил и подтвердил.

Основные контексты:

- утреннее планирование;
- фиксация дел в дороге;
- выгрузка мыслей после встречи;
- бытовой список и рабочие поручения в одном сообщении;
- восстановление контроля в перегруженный день.

## Термины и границы

| Операция | Что означает | Правило MVP |
| --- | --- | --- |
| Сегментация | Выделить независимые дела из одной фразы | Входит |
| Нормализация | Сделать короткие названия без потери смысла | Входит |
| Приоритизация | Предложить один главный фокус и объяснить причину | Входит как предложение |
| Группировка | Показать совместимые дела рядом | Входит только в preview |
| Объединение | Слить истинные дубли | Только через отдельное подтверждение `SMART-010` |
| Декомпозиция | Разбить одно дело на этапы | Не входит, это `SMART-013` |
| Планирование времени | Назначить длительность и слот | Не входит, это `CAL-002` |
| Маршрут | Объединить дела по географии | Не входит без адресов, согласий и отдельного решения |

Связанные, но разные задачи нельзя физически склеивать. «Позвонить поставщику» и «написать Юре» могут находиться в одном коммуникационном блоке, но сохраняются как две записи.

## Основной UX-сценарий

### 1. Голосовой ввод

Пользователь говорит:

```text
Сегодня нужно позвонить поставщику, закончить отчёт,
записаться к врачу, забрать заказ и не забыть написать Юре.
```

### 2. Распознавание и анализ

Экран показывает последовательные состояния:

```text
Распознаю речь → Разбираю дела → Собираю план
```

До подтверждения ничего не сохраняется.

### 3. Превью

Эталонный ответ:

```text
Я услышал 5 дел. Сроки не ставлю — ты их не называл.

Предлагаю такой порядок:

Главный фокус
• Закончить отчёт.

Короткий коммуникационный блок
• Позвонить поставщику.
• Написать Юре.
Сохраню как две отдельные задачи.

Остальные дела
• Записаться к врачу.
• Забрать заказ.

Отчёт ставлю первым как наиболее объёмное дело. Это моя оценка,
а не факт из твоей фразы.

[Добавить 5 задач] [Выбрать другой фокус] [Добавить без приоритета]
```

### 4. Действия пользователя

- `Добавить 5 задач` — создать пять отдельных задач; выбранный фокус получает `priority: high`, остальные сохраняются без выдуманных сроков.
- `Выбрать другой фокус` — показать пять компактных кнопок/строк; после выбора обновить preview, но ещё ничего не сохранять.
- `Добавить без приоритета` — создать пять задач с обычным приоритетом.
- Редактирование карточки — изменить текст, исполнителя, срок или удалить пункт до подтверждения.
- `Отмена` — закрыть preview без записи.

Кнопка `Да` и абстрактная кнопка `Ответить` запрещены: CTA должен называть результат действия.

## Политика уточняющих вопросов — SMART-009 v2

SMART-014 использует механику одного вопроса, но меняет правило его появления.

### Задавать вопрос только когда

Неопределённость существенно меняет создаваемые данные или смысл плана:

1. Непонятно, одна это сложная задача или несколько независимых.
2. В речи есть противоречащие друг другу сроки.
3. Неясный исполнитель может поставить задачу другому реальному человеку.
4. Пользователь явно просит объединить записи, но найдено несколько возможных дублей.
5. Без ответа невозможно безопасно выполнить запрошенное действие.

### Не задавать вопрос когда

- срок не назван — сохранить `deadline: null`;
- исполнитель не назван — использовать текущего пользователя;
- длительность неизвестна — не выдумывать её;
- пользователь сказал «записаться к врачу» — не превращать это в поездку;
- AI может предложить фокус и дать кнопку изменения без блокирующего вопроса;
- сомнение влияет только на удобство, но не на корректность данных.

Максимум один вопрос. Всегда должна быть неблокирующая кнопка `Сохранить как есть`, если это безопасно.

## Правила AI

### Сегментация

1. Выделять отдельные действия по глаголам, объектам, исполнителям, срокам и зависимостям.
2. Конструкцию «не забыть написать Юре» трактовать как положительную задачу `Написать Юре`.
3. Не разделять единое действие только из-за нескольких объектов: «собрать и отправить отчёт» может быть одной задачей, если это один результат.
4. Если фраза описывает проект с этапами, вернуть одну задачу и предложить `SMART-013`, а не создавать много независимых задач автоматически.
5. Сохранять исходную формулировку в `originalMsg` для трассировки.

### Факты и предположения

Каждое значимое поле получает источник:

- `explicit` — прямо сказано пользователем;
- `context` — взято из подтверждённой задачи или профиля;
- `inferred` — предложение AI, которое нельзя записывать без подтверждения.

Запрещено без явных данных:

- создавать дедлайн `16:00`;
- утверждать длительность `10 минут`;
- считать запись к врачу поездкой;
- говорить о свободном окне календаря;
- добавлять адрес, исполнителя или обязательство;
- выдавать предположение за факт.

### Приоритет

Порядок сигналов:

1. Явная срочность и срок пользователя.
2. Уже сохранённый deadline/priority существующей задачи.
3. Зависимость: действие разблокирует другие дела.
4. Явно названный пользовательский фокус.
5. Только затем — объяснимое предложение AI.

Если используются только мягкие признаки, формулировка должна быть: `Предлагаю поставить первым...`, а не `Главное сегодня...` как установленный факт.

В эталонном сценарии отчёт можно предложить первым как вероятно более объёмное дело, но это предположение должно быть обозначено и легко изменяться одной кнопкой.

## Контракт ответа AI

Рекомендуемый транспортный блок для текущего `/anthropic`:

```text
<task_plan>{...валидный JSON...}</task_plan>
```

Служебный блок не показывается пользователю. Приложение рендерит UI только после схемной валидации.

```json
{
  "version": 1,
  "mode": "multi_task_plan",
  "transcript": "Сегодня нужно позвонить поставщику...",
  "tasks": [
    {
      "tempId": "t1",
      "text": "Позвонить поставщику",
      "person": "Я",
      "deadline": null,
      "priority": "normal",
      "source": {
        "text": "explicit",
        "person": "default_current_user",
        "deadline": "missing"
      },
      "confidence": 0.97,
      "originalFragment": "позвонить поставщику"
    }
  ],
  "plan": {
    "focusTaskTempId": "t2",
    "focusReason": "Предположительно наиболее объёмное дело",
    "focusReasonSource": "inferred",
    "groups": [
      {
        "id": "communication",
        "title": "Короткий коммуникационный блок",
        "taskTempIds": ["t1", "t5"],
        "persistAsGroup": false
      }
    ]
  },
  "clarification": null,
  "warnings": [
    "Сроки не указаны и не будут добавлены"
  ]
}
```

Если вопрос действительно нужен:

```json
{
  "clarification": {
    "kind": "task_boundary",
    "question": "Это одно дело или два отдельных?",
    "options": [
      {"id": "one", "label": "Одна задача"},
      {"id": "two", "label": "Две задачи"},
      {"id": "as_is", "label": "Сохранить как есть"}
    ]
  }
}
```

### Валидация

- `tasks`: от 1 до 12 элементов;
- уникальный `tempId`;
- `text`: 1–140 символов;
- `deadline`: `null` или валидное значение поддерживаемого формата;
- `priority`: `low | normal | high`;
- `confidence`: число 0–1;
- все `taskTempIds` должны существовать;
- `focusTaskTempId` либо существует, либо `null`;
- неизвестные поля отбрасываются;
- невалидный ответ не изменяет задачи и показывает повторную попытку.

## Технический план

### Frontend / Mini App

Точки интеграции в актуальном `index.html`:

- `sendVoiceMessage()` — вместо немедленного сохранения длинного task intent запускать анализ плана;
- `fallbackTaskFromText()` — оставить fallback для простой одиночной задачи при недоступности AI;
- `createTaskFromChat()` / `saveTaskWithDedup()` — переиспользовать нормализацию и дедупликацию на уровне каждой подтверждённой задачи;
- `SMART-008` action preview — переиспользовать принцип явного подтверждения, но сделать отдельный multi-task UI;
- `SMART-009` clarification UI — переиспользовать кнопки, изменив критерий появления вопроса;
- `SMART-010` — не запускать серию блокирующих `window.confirm()` для пяти задач; собрать найденные дубли в одном preview.

Предлагаемые функции:

```text
requestVoiceTaskPlan(transcript)
parseVoiceTaskPlan(response)
validateVoiceTaskPlan(plan)
renderVoiceTaskPlanPreview(plan)
updateVoiceTaskPlanDraft(change)
selectVoiceTaskPlanFocus(tempId)
confirmVoiceTaskPlan(mode)
saveVoiceTaskBatch(tasks, idempotencyKey)
```

Новый интерфейс должен быть dialog/bottom sheet на mobile, с корректным focus trap, Escape/Назад, touch targets не меньше 44×44 и без перекрытия нижней навигацией.

### Worker

Worker находится в отдельном репозитории. Перед реализацией подключить его checkout и обновить `FILE_MAP_WORKER.md` по реальному коду.

Рекомендуемый endpoint:

```text
POST /tasks/batch
Authorization: текущая пользовательская сессия
Idempotency-Key: UUID одной подтверждённой операции
```

Request:

```json
{
  "source": "voice_multi_task",
  "originalTranscript": "...",
  "tasks": [
    {"clientId": "t1", "text": "Позвонить поставщику", "deadline": null, "priority": "normal"}
  ]
}
```

Response:

```json
{
  "ok": true,
  "idempotencyKey": "...",
  "created": [{"clientId": "t1", "taskId": "..."}],
  "merged": [],
  "errors": []
}
```

Требования:

1. Повтор с тем же ключом не создаёт дубли.
2. Сервер валидирует максимум 12 задач и все поля.
3. Операция либо сохраняет весь набор, либо возвращает понятный результат частичного сбоя; предпочтителен атомарный D1 transaction.
4. Если текущая task storage-модель не позволяет транзакцию, Worker хранит operation record и обеспечивает безопасный retry по `clientId`.
5. AI-анализ не имеет права напрямую вызывать запись задач.
6. Аудиофайл не хранится после транскрипции без отдельного согласия.

### Fallback

- AI недоступен: показать транскрипцию и предложить `Создать одной задачей` или `Попробовать ещё раз`.
- Невалидный JSON: ничего не сохранять.
- Batch save не удался: сохранить draft локально до повторной попытки; не повторять запрос автоматически без idempotency key.
- Низкая уверенность отдельного пункта: подсветить только этот пункт для редактирования, не блокировать весь список.

События:

```text
voice.multiTask.detected
voice.multiTask.previewShown
voice.multiTask.clarificationShown
voice.multiTask.focusChanged
voice.multiTask.taskEdited
```

### Метрика ценности

Основная:

```text
доля multi-task voice сессий,
где пользователь подтвердил 2+ задачи не более чем с одной правкой
```

Дополнительные:

- корректное количество задач в тестовом наборе;
- доля подтверждений preview;
- среднее число правок;
- время от окончания записи до сохранения;
- повторное использование функции за 7 дней;
- доля созданных задач, закрытых за 7 дней;
- доля пользователей, выбравших другой фокус;
- стоимость AI-вызова и p95 latency.

Guardrails:

- 0 незапрошенных автоматических записей;
- 0 выдуманных дедлайнов в эталонном наборе;
- 0 повторных задач при retry с тем же idempotency key;
- отсутствие аудиотекста и персональных данных в lite analytics.

## Проверка гипотезы до разработки

Провести быстрый moderated usability test на 5–8 пользователях:

1. Попросить назвать 3–6 реальных дел одной фразой.
2. Показать интерактивный или Wizard-of-Oz preview.
3. Не объяснять назначение кнопок заранее.
4. Проверить, понимает ли человек, что сохранится.
5. Спросить, доверяет ли он предложенному фокусу и почему.

Целевые гипотезы:

- не менее 80% участников понимают результат CTA без подсказки;
- не менее 80% успешно подтверждают или меняют фокус;
- не менее 70% считают сценарий полезнее обычной голосовой диктовки;
- ни один пользователь не считает, что связанные задачи будут слиты в одну;
- критические ошибки доверия отсутствуют.

Эти числа являются стартовыми порогами discovery, а не доказанными метриками продукта.

## Acceptance criteria MVP

1. Одна голосовая фраза может дать от 2 до 12 отдельных задач.
2. До подтверждения ни одна задача не записывается.
3. Пользователь видит точное количество распознанных дел.
4. Каждую задачу можно изменить или удалить в preview.
5. AI предлагает максимум один главный фокус и объясняет основание.
6. Предположение AI визуально отличается от факта пользователя.
7. Отсутствующий срок остаётся `null` и не вызывает обязательного вопроса.
8. `Записаться к врачу` не превращается в поездку или встречу.
9. Связанные задачи сохраняются отдельно.
10. Истинные дубли проходят отдельное подтверждение объединения.
11. CTA явно сообщает результат: `Добавить N задач`.
12. `Добавить без приоритета` не назначает `high` ни одной задаче.
13. `Выбрать другой фокус` меняет только draft до сохранения.
14. Повтор batch-запроса не создаёт дубли.
15. Ошибка AI или сети не приводит к частичной скрытой записи.
16. Preview работает на узком mobile viewport и в Telegram WebView.
17. Voice consent и privacy-поведение не регрессируют.
18. Существующий single-task voice flow продолжает работать.

## Эталонные QA-кейсы

| Вход | Ожидаемое поведение |
| --- | --- |
| `Позвонить маме завтра` | Одна задача, явный срок, без multi-task плана |
| Исходный пример с пятью делами | Пять задач, без сроков, отчёт предложен как изменяемый фокус |
| `Собрать данные и отправить отчёт клиенту` | Одна задача либо один вопрос о границе, не два автоматических пункта |
| `Позвонить Юре и после звонка отправить ему файл` | Две задачи с зависимостью, не один дубль |
| `Не забыть купить молоко и не звонить поставщику` | Купить молоко; отрицательное действие не превращается в задачу без явного намерения |
| `Отчёт сегодня и отчёт в пятницу` | Один вопрос о противоречащем сроке либо две явно различённые задачи |
| `Записаться к врачу и забрать заказ` | Две задачи; никаких поездок, маршрута и свободного окна |
| Повторить подтверждение после timeout | Те же task IDs, без дублей |
| AI вернул сломанный JSON | Ошибка preview, задачи не изменены |
| Пользователь нажал `Добавить без приоритета` | Все задачи `normal`, порядок AI не превращён в сохранённый факт |

Полный эталонный ответ также фиксируется в `pm/assistant-evaluation.md` как regression для `SMART-009` и `SMART-014`.

## Что не входит в MVP

- автоматические дедлайны;
- оценка длительности как сохранённый факт;
- автоматический time blocking;
- чтение свободных окон внешнего календаря;
- маршрутизация и геолокация;
- вложенные подзадачи;
- физическое хранение временных групп плана;
- фоновое прослушивание;
- автоматическое объединение дублей;
- изменение существующих задач без preview;
- голосовой режим без подтверждения обработки голоса.

## Риски

| Риск | Снижение |
| --- | --- |
| AI неверно определил количество задач | Редактируемые карточки, confidence, эталонный набор |
| Убедительный, но выдуманный план | Источники `explicit/context/inferred`, запрет на автосроки |
| Слишком много вопросов | Только material uncertainty, максимум один вопрос |
| Потеря скорости голосового ввода | Замер latency, быстрый single-task fallback |
| Частичное сохранение пяти задач | Batch endpoint, idempotency, operation record |
| Серия раздражающих confirm по дублям | Единый duplicate-review в preview |
| Scope creep в календарь | Жёсткие non-goals и зависимость от CAL-002 |
| Пользователь не понимает последствия CTA | Точные глагольные кнопки и usability test 5–8 человек |
| Утечка чувствительного текста | Не писать transcript/audio в lite analytics, соблюдать voice consent |

## Этапы после MVP

### Этап 2 — умный план без календаря

- учитывать подтверждённые дедлайны и приоритеты существующих задач;
- показывать зависимости;
- объяснять риск перегруза только как качественное предположение;
- учитывать явные пользовательские правила из AI-памяти.

### Этап 3 — связка с CAL-002

- `durationMin`;
- внутренние planned slots;
- `fits / tight / overloaded`;
- ручное подтверждение time blocking;
- без записи во внешний календарь.

### Этап 4 — контекст места

Только после отдельного product/privacy решения: адреса, opt-in геолокация и предложения маршрута. Не является продолжением MVP по умолчанию.

## Definition of Done

SMART-014 можно перевести в Done, когда:

1. Утверждено решение Алексея о старте после beta gate.
2. Успешно пройдён usability test или согласован осознанный пропуск discovery.
3. Frontend показывает валидируемый multi-task preview.
4. Worker сохраняет batch идемпотентно.
5. Эталонные и негативные QA-кейсы проходят.
6. Пройден живой smoke в Telegram Mini App минимум на iOS и Android.
7. Аналитика считает funnel без текста задач и аудио.
8. Обновлены `FILE_MAP_UI.md`, `FILE_MAP_WORKER.md`, QA и рабочие логи.
9. Результат возвращён с ветками/коммитами app и Worker и статусом ручной проверки.
