# BACK-055 — Уведомления как лента внимания

**Дата:** 2026-07-08
**Владелец:** Юрий
**Тип:** Product/UI
**Приоритет:** P1
**Статус:** Done
**Рекомендуемая ветка:** `feat/notifications-action-cards`

## Контекст

Сейчас экран `notifications` выглядит как обычный архив уведомлений: фильтры есть, но при пустом состоянии пользователь видит только `Нет уведомлений этого типа`. Для продукта 4 это слабый сценарий, потому что позиционирование уже не "таск-менеджер", а AI-планер / персональный операционный директор.

Решение: сделать уведомления не архивом сообщений, а **лентой внимания**. Уведомление должно отвечать на три вопроса:

```text
Что произошло?
Почему это важно?
Что я могу сделать сейчас?
```

Почему решили делать именно так:

1. **Пользовательский смысл:** человеку не нужно читать список событий; ему нужно быстро решить, что делать с задачей.
2. **Связь с AI-планером:** главный экран уже отвечает "что важно сегодня", а уведомления должны продолжать этот же сценарий в течение дня.
3. **Меньше перегруза:** не дублируем всю карточку задачи внутри уведомления, а даём быстрый мост к нужному действию.
4. **Безопасный MVP:** текущий экран уже умеет раскрывать уведомления и показывать действия, поэтому можно развить существующий компонент, а не переписывать экран с нуля.

## Визуальный ориентир

Reference-макет: `docs/tasks/assets/BACK-055-notifications-action-cards-wireframe.svg`

Это не пиксельный макет. Использовать как направление: тёмный фон, компактные glass-карточки, зелёный акцент, короткие действия, без hero-блока и без маркетингового оформления.

Важно: на карточке должна быть кнопка `Отложить`, а не отдельная кнопка `+1 час`. Вариант `1 час` показывается только внутри раскрытого меню `Отложить`.

## Что сделать в текущем прокате

### 1. Переименовать вкладку

В фильтрах экрана `notifications` заменить пользовательский текст:

```text
Дедлайны -> Горит
```

Важно: внутренний тип можно оставить `deadline`, если так проще и не требуется backend-миграция. Меняем смысл для пользователя, не обязательно модель данных.

Итоговые вкладки:

```text
Все / Задачи / Горит / Напоминания / Система
```

### 2. Обновить пустые состояния

Не показывать сухой текст `Нет уведомлений этого типа`.

Для вкладки `Все`:

```text
Пока всё спокойно
4 следит за задачами, дедлайнами и напоминаниями. Когда появится важное событие — оно будет здесь.
```

Для вкладки `Горит`:

```text
Нет горящих задач
Сейчас нет просрочек и дедлайнов на ближайшее время.
```

Для остальных вкладок можно использовать короткий вариант:

```text
Пока нет событий
Когда появится важное действие, 4 покажет его здесь.
```

### 3. Сделать единую структуру карточки уведомления

Карточка должна быть короткой и предсказуемой:

```text
[тип / срочность]        [время]

Заголовок
Короткое описание

[главное действие] [второе действие] [ещё]
```

Пример свернутой карточки:

```text
Горит дедлайн
Через 45 минут: отправить тестовый отчёт

[К задаче] [Отложить]
```

Пример раскрытого блока `Отложить`:

```text
Отложить на:
[15 мин] [1 час] [3 часа] [Завтра]
```

Не показывать все варианты отложить сразу на каждой карточке. Сначала пользователь видит кнопку `Отложить`, варианты появляются только после нажатия.

### 4. Добавить действия по типам уведомлений

| Тип уведомления | Основные действия |
| --- | --- |
| `task` | `К задаче`, `Готово` |
| `deadline` / вкладка `Горит` | `К задаче`, `Отложить`, `Готово` |
| `reminder` | `Готово`, `Отложить`, `К задаче` |
| `system` | `Понятно` или переход в нужный экран, если он реально нужен |
| `waiting` / "Ждёшь ответ" если есть такой payload | `Написать`, `Открыть задачу` |

Если отдельного типа `waiting` пока нет, не вводить большой backend-рефакторинг. Можно оставить это как future-ready обработку: если в уведомлении есть `assigneeUsername` / `assigneeTgId` / `person`, показывать действие `Написать`, иначе не показывать.

### 5. Реализовать `Отложить`

Кнопка `Отложить` раскрывает компактное меню:

```text
15 мин
1 час
3 часа
Завтра
```

Поведение:

1. Для уведомления с задачей обновить срок/напоминание задачи.
2. После выбора показать toast: `Отложено`.
3. Уведомление можно отметить прочитанным или обновить его состояние локально.
4. Если backend пока не поддерживает точное отложение уведомления, минимум должен обновляться срок/напоминание связанной задачи через существующий task update flow.

### 6. Реализовать настоящий переход `К задаче`

Сейчас `notifGoToTask()` фактически возвращает на Home. Нужно, чтобы действие открывало конкретную карточку задачи.

Требование к payload уведомления:

```json
{
  "id": "...",
  "type": "deadline",
  "taskId": "...",
  "title": "Горит дедлайн",
  "detail": "Через 45 минут: отправить тестовый отчёт",
  "ts": 1234567890
}
```

Допустимые имена поля для совместимости: `taskId`, `task_id`, `relatedTaskId`. Но внутри фронта лучше нормализовать в одно значение.

Если `taskId` нет:

1. Не показывать кнопку `К задаче`; или
2. Показывать disabled-состояние нельзя, лучше скрыть кнопку.

### 7. Связать уведомления с текущей карточкой задачи

Текущая карточка задачи уже содержит:

- заголовок;
- описание;
- срок;
- быстрые сроки `Сегодня / Завтра / Неделя`;
- статус `Активная / Горит / Пауза / Готово`;
- приоритет;
- напоминание;
- направление;
- человек / исполнитель;
- чек-лист;
- обсуждение;
- историю;
- совет от 4.

Уведомления должны быть быстрым входом в эти поля:

| Уведомление | Поле карточки задачи | Что делает пользователь |
| --- | --- | --- |
| `Горит дедлайн` | `Срок`, `Статус`, `Приоритет` | Открывает задачу, завершает или откладывает |
| `Напоминание` | `Напоминание`, `Срок` | Завершает, откладывает или открывает задачу |
| `Ждёшь ответ` | `Человек`, `Обсудить задачу` | Пишет исполнителю или открывает задачу |
| `Задача изменилась` | `История` | Позже можно открыть историю изменений |
| `Сложная задача` | `Чек-лист`, `Совет от 4` | Позже можно спросить 4 или разбить на этапы |

## Учесть архитектурно, но не делать сейчас

Не реализовывать в текущем прокате, но не закрывать архитектурно:

1. `Спросить 4` по задаче.
2. `Разбить на этапы`.
3. `Посмотреть историю изменений`.
4. Умные варианты отложить по времени суток:
   - утром: `15 мин / 1 час / После обеда / Завтра`;
   - днём: `30 мин / 1 час / Вечером / Завтра`;
   - вечером: `Завтра утром / Завтра днём / На неделю`.
5. Дедупликация похожих задач.

## Не делать в текущем прокате

- Полное редактирование задачи внутри уведомления.
- Комментарии прямо внутри уведомления.
- Длинные AI-объяснения в карточке.
- Больше 3 основных кнопок на карточке.
- Сложный календарь переноса внутри уведомления.
- Большой backend-рефакторинг типов уведомлений, если хватает текущих `task/deadline/reminder/system`.

## Проверка

1. Открыть экран `Уведомления` на mobile viewport `390x844`.
2. Проверить вкладки: `Все / Задачи / Горит / Напоминания / Система`.
3. Проверить, что пустое состояние не звучит как ошибка.
4. Создать или подложить тестовые уведомления типов `task`, `deadline`, `reminder`, `system`.
5. Проверить, что `К задаче` открывает конкретный `task-detail`, а не просто Home.
6. Проверить, что `Готово` завершает связанную задачу и обновляет список.
7. Проверить, что `Отложить` раскрывает варианты `15 мин / 1 час / 3 часа / Завтра`.
8. Проверить, что выбор отложить обновляет срок/напоминание задачи и не ломает карточку.
9. Проверить, что карточки не распирают экран и кнопки не налезают друг на друга.
10. Проверить, что на уведомлениях без `taskId` не показываются действия, которые невозможно выполнить.

## Критерий готовности

Пользователь открывает уведомления и за 1-2 тапа может:

- понять, что сейчас горит;
- открыть конкретную задачу;
- завершить задачу;
- отложить задачу на короткий срок;
- перейти к исполнителю, если у задачи есть связанный человек;
- не видеть пустой или декоративный экран без следующего действия.
