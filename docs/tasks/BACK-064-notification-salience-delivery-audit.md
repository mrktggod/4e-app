# BACK-064 - Notification salience and delivery audit

**Дата:** 2026-07-21
**Владелец:** Codex / Yuri / Алексей QA
**Тип:** Bug / QA / Product reliability
**Приоритет:** P1
**Статус:** Triaged

## Контекст

Алексей сообщил: уведомления в продукте есть, но их плохо видно и они иногда не срабатывают. Нужно не только проверить экран `notifications`, а пройти весь путь сигнала внимания: событие -> backend delivery -> Telegram/Mini App/user-visible surface -> звук/вибрация/видимый badge/card -> действие пользователя.

`BACK-055` закрыл ленту уведомлений и action cards. `BACK-017` описывает delivery smoke для Telegram. Этот task закрывает оставшийся слой: надежность, заметность и сенсорный сигнал.

## Что проверить

1. Доставка:
   - утренний брифинг;
   - просроченная задача;
   - напоминание по задаче;
   - новая/назначенная задача, если payload уже поддерживается;
   - системное уведомление, если оно есть в `/notifications`.

2. Видимость:
   - badge/точка на колокольчике появляется без ручного открытия экрана;
   - экран `notifications` показывает свежую карточку сверху;
   - unread-состояние заметно;
   - пустые состояния не маскируют реальные ошибки загрузки;
   - ошибка загрузки `/notifications` не выглядит как "уведомлений нет".

3. Сенсорный сигнал:
   - в Telegram приходит нативное уведомление со звуком, если звук включен в клиенте;
   - на телефоне есть вибрация, если она разрешена системой/Telegram;
   - внутри Mini App для новых важных уведомлений используется доступная вибрация через Telegram WebApp/HapticFeedback или `navigator.vibrate`, если платформа разрешает;
   - если звук/вибрация технически недоступны в WebView, это явно зафиксировано как platform limitation.

4. Действия:
   - `К задаче` открывает конкретную задачу;
   - `Готово` завершает связанную задачу;
   - `Отложить` переносит срок/напоминание;
   - `Понятно` снимает unread;
   - после действия badge и unread-счетчик синхронизируются.

## Acceptance criteria

- На staging есть воспроизводимый smoke для минимум двух типов: reminder/deadline и morning briefing.
- Уведомление реально приходит правильному пользователю, а не только появляется в `/notifications`.
- Пользователь видит сигнал без похода в архив: badge, карточка, Telegram-native notification или другой явный surface.
- Для mobile/TMA зафиксирован результат по звуку и вибрации: работает / не работает из-за ограничений платформы / нужна отдельная реализация.
- Если backend delivery не срабатывает, создан отдельный root-cause bug с endpoint/route/log evidence.
- Если delivery работает, но сигнал незаметен, создан UI/UX fix scope по badge, haptic и foreground alert.

## Первичные code-read наблюдения

- `scripts/task-ui-renderers.js` загружает экран через `openNotifications()` -> `/notifications`, затем обновляет `notifCache` и `bell-dot`.
- `BACK-055` action cards уже проверяли headless smoke, но он не доказывает real delivery, звук или вибрацию.
- В коде есть `navigator.vibrate` для task-card swipe, но нет отдельного явного notification haptic layer.
- `BACK-017` уже описывает ручной Telegram delivery smoke; его нужно расширить проверкой заметности.

## Не делать в этом task без отдельного решения

- Не включать массовые production-уведомления.
- Не менять цены, entitlement, платежи или auth.
- Не слать реальные уведомления beta-пользователям без согласованного окна QA.
- Не считать source-only проверку достаточной для Done.
