# BACK-048 — Тестовые аккаунты разработчиков с полными правами

**Приоритет:** P1
**Ответственный:** Юрий / worker
**Ветка:** `feat/dev-test-accounts`
**Одна задача за сессию**

## Контекст

Перед закрытым тестом команде нужны стабильные dev/QA-аккаунты, чтобы проверять продукт без ручных оплат, случайных trial-истечений и смешивания с реальными пользователями. Сейчас регистрация даёт обычный trial/Premium, а карта `BACK-040` на Admin API управления тарифами ещё не является готовым инструментом.

## Что нужно сделать

### 1. Создать безопасный способ заведения test/dev аккаунтов

В worker-репозитории добавить один из вариантов:

- защищённый seed/admin action, который создаёт или обновляет dev-аккаунт по email;
- или расширить будущий `/admin/users/:id/plan` из `BACK-040`, чтобы можно было выдать full-access/Premium конкретному test-user.

Обязательное условие: доступ только через `ADMIN_SECRET` или другой server-side secret. Нельзя делать публичный endpoint, который любой пользователь сможет вызвать из браузера.

### 2. Набор аккаунтов

Минимальный набор:

| Роль | Назначение |
| --- | --- |
| Алексей dev | Проверка продукта владельцем и PM/QA-сценарии |
| Юрий dev | Проверка backend/bot/маркетинговых сценариев |
| QA/dev общий | Повторяемый smoke без привязки к личному аккаунту |

Точные email и пароли передать Алексею вне git: Telegram/личное сообщение/секретный менеджер. В репозитории не хранить реальные пароли, токены, session id и `ADMIN_SECRET`.

### 3. Права аккаунтов

Каждый dev-аккаунт должен иметь:

- активный Premium/full-access без реальной оплаты;
- доступ к созданию/редактированию задач;
- доступ к AI-чату;
- доступ к голосовому вводу;
- доступ к subscription/payment screens без блокировки истёкшим trial;
- отдельную метку/признак test/dev, если такая модель уже есть или её легко добавить без большого рефакторинга.

### 4. Окружения

Сначала сделать в staging. В production добавлять только если Алексей подтверждает, что эти аккаунты нужны для live smoke.

Для production-аккаунтов обязательно:

- не смешивать их с реальными пользователями;
- использовать очевидные test/dev email;
- не включать реальные платежные операции;
- иметь ручной способ отключить/удалить аккаунты.

### 5. Тестовые данные

Для каждого аккаунта добавить небольшой seed-набор:

- 2-3 активные задачи;
- 1 просроченная задача;
- 1 задача на сегодня;
- 1 завершённая задача, если модель completed уже поддерживается.

Это нужно, чтобы Home, task detail, AI-планер, статистика и списки не открывались пустыми.

## Что не делать

- Не коммитить реальные пароли, токены, ключи и `ADMIN_SECRET`.
- Не делать bypass оплаты доступным из фронта без секрета.
- Не выдавать full-access всем новым пользователям.
- Не ломать обычный trial/paid flow.
- Не удалять и не перезаписывать реальные пользовательские данные.

## Критерий готовности

- В staging есть минимум 3 dev/test аккаунта с Premium/full-access.
- Вход по email/password проходит на web/VK/TG surface, где применимо.
- `/auth/me` или экран подписки показывает активный Premium/full-access.
- Создание, редактирование и завершение задачи работают.
- AI-чат отвечает на базовый запрос.
- Payment/subscription screen не блокирует сценарий из-за истёкшего trial.
- Пароли переданы Алексею вне репозитория.
- В `shared/WORK_LOG.md` указано, где созданы аккаунты: staging, production или оба.

## Проверка после выполнения

1. Login каждым dev-аккаунтом.
2. Открыть Home: видны seed-задачи и нет пустого состояния.
3. Создать новую задачу через quick-add.
4. Открыть task detail и изменить срок/статус.
5. Открыть AI-чат и задать вопрос по задачам.
6. Открыть subscription/payment: Premium активен, trial не истёк.
7. Проверить, что секреты не попали в `git diff`.
