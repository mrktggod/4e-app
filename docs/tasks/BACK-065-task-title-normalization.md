# BACK-065 - Task title normalization from dictated intent

**Дата:** 2026-07-21
**Владелец:** Codex / Yuri
**Тип:** Product/AI/UX
**Приоритет:** P2
**Статус:** Ready for QA

## Контекст

Алексей заметил, что заголовки задач иногда получаются нечитаемыми, потому что в название попадает сырая фраза из диктовки или чата. Пример: вместо нормального заголовка появляется что-то вроде "я придумать на платформе". Нужно, чтобы 4 помогал формировать читаемый заголовок задачи.

## Проблема

Сейчас `fallbackTaskFromText()` чистит только часть служебных слов (`поставь`, `добавь`, `надо`, `нужно`) и затем `formatTaskIntentTitle()` просто капитализирует строку. `createTaskFromChat()` сохраняет `task.text` как основной заголовок, а `originalMsg` хранит исходный запрос.

Этого мало для живой речи:

- фраза может начинаться с "я", "мне", "тебе надо", "надо тебе", "давай";
- инфинитивы и оборванные фразы остаются кривыми;
- обращение к ассистенту смешивается с названием задачи;
- длинный контекст должен уйти в описание/originalMsg, а не в заголовок.

## Цель

Сделать так, чтобы у задачи был короткий, читаемый, самостоятельный заголовок:

```text
Сырой ввод: "так, тебе надо придумать на платформе как вести разработку агентабота через задачи"
Заголовок: "Придумать управление разработкой агентабота через задачи"
Описание/originalMsg: полный исходный текст пользователя
```

## Acceptance criteria

- `task.text` становится коротким заголовком 3-9 слов, без мусорных вводных и случайного "я/мне/тебе надо".
- `originalMsg` сохраняет полную исходную фразу без потерь.
- Для ручного quick-add короткая нормальная фраза не портится.
- Для AI-chat и voice task creation нормализация применяется одинаково.
- Если заголовок неоднозначный, 4 предлагает вариант и позволяет поправить через уже существующий edit title flow.
- Заголовок не должен менять смысл: исполнитель, дедлайн и описание не теряются.

## Примеры

| Сырой ввод | Ожидаемый заголовок |
| --- | --- |
| "мне надо позвонить Ивану завтра" | "Позвонить Ивану" |
| "тебе надо придумать на платформе как вести разработку агентабота через задачи" | "Придумать управление разработкой агентабота через задачи" |
| "я придумать на платформе уведомления для агентов" | "Придумать уведомления для агентов" |
| "поставь Маше задачу подготовить договор к пятнице" | "Подготовить договор" |
| "надо проверить почему пропадает часть чата" | "Проверить пропажу части чата" |

## Возможная реализация

1. Добавить отдельный helper `normalizeTaskTitle(rawText, context)`:
   - удаляет вводные фразы;
   - удаляет обращение к ассистенту;
   - выносит дедлайн/исполнителя в поля, если они уже распознаны;
   - ограничивает длину;
   - приводит первую букву к верхнему регистру.

2. Для AI-chat попросить модель возвращать не только `text`, но и `originalMsg`/`description`:

```json
{
  "text": "Придумать управление разработкой агентабота через задачи",
  "originalMsg": "так, тебе надо придумать на платформе как вести разработку агентабота через задачи"
}
```

3. Для fallback без AI использовать эвристику, а не только capitalization.

4. В подтверждении задачи показывать заголовок отдельно от исходного текста, чтобы пользователь видел, что 4 понял.

- Не менять модель данных задач без необходимости.
- Не удалять `originalMsg`.
- Не делать длинную анкету перед созданием задачи.
- Не ломать уже существующий edit-title flow в task-detail.

## Implementation 2026-07-22

- Добавлен `normalizeTaskTitle(rawText, context)` для короткого заголовка без вводных вроде `я`, `мне надо`, `тебе надо`, `поставь Маше задачу`.
- `fallbackTaskFromText()` теперь сохраняет `originalMsg`, извлекает deadline-фразы на кириллических пробельных границах и не держит срок в `task.text`.
- `createTaskFromChat()`, voice task creation и quick-add используют один helper; quick-add сохраняет сырой ввод в `originalMsg`, если заголовок был нормализован.
- AI-chat system prompt теперь просит `<create_task>` возвращать короткий `text` и полный `originalMsg`.
- Добавлен `npm run smoke:back065`, который проверяет примеры из этого task-дока.
