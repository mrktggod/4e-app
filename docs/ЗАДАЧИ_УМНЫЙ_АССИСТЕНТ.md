# Задачи SMART: умное взаимодействие бота и приложения

Дата: 2026-07-04. Основано на анализе кода: `4e-worker/src/bot/handler.js`, `analyzer.js`, `reminders.js`, `worker.js`, `4e-app/index.html` (AI-чат ~строки 4326–4423, system prompt ~2819).

**Что уже есть (фундамент):** бот видит групповые чаты и сохраняет сообщения в KV; AI-парсинг задач через Claude Haiku с полем `person` (имя исполнителя текстом); intent-детект в приложении; AI-чат получает историю (40 сообщений) + 12 активных задач; напоминания каждые 15 мин, проверка дедлайнов каждый час; функция утреннего брифинга `checkBriefings()` написана, **но не подключена**.

**Чего нет:** участники чата не отслеживаются, mention'ы не парсятся, исполнитель не связан с TG ID, подтверждения многословные, AI-чат не знает имени пользователя и не помнит ничего между сессиями.

## Блок 1. Групповой секретарь

| # | Задача | Приоритет |
|---|---|---|
| SMART-001 | **Ростер участников чата** | P1 |
| SMART-002 | **Исполнитель с TG ID в задаче** | P1 |
| SMART-003 | **Кнопка «Написать» у исполнителя** | P1 |
| SMART-004 | **Лаконичная фиксация в группах** | P1 |
| SMART-005 | **Утренняя сводка по чату** | P2 |

**SMART-001 — Ростер участников чата.**
Бот копит карту участников per chat: `chat_id → [{tg_id, first_name, last_name, username, last_seen}]`. Источники: `msg.from` каждого сообщения (уже проходят через handler.js), события `new_chat_members`/`left_chat_member` (сейчас не обрабатываются), `reply_to_message.from`. Хранить в D1.
*Зачем:* Bot API не даёт списка участников группы — узнать «кто в чате по именам» можно только накапливая. Это фундамент для SMART-002/003/005.
*Техническое:* убедиться, что privacy mode у бота отключён (судя по тому, что он сохраняет все сообщения — уже так). Файл: `handler.js`.

**SMART-002 — Исполнитель с TG ID в задаче.**
Три источника привязки `person → tg_id`: 1) entities сообщения: `text_mention` даёт tg_id напрямую, `mention` (@username) резолвится через ростер; 2) fuzzy-матч имени из AI-парсинга (`result.person`) против ростера («Вася» → Василий Петров); 3) если задача из reply — автор исходного сообщения кандидат в исполнители. В объект задачи добавить `assigneeTgId`, `assigneeUsername`; передавать список участников в промпт analyzer.js, чтобы Haiku выбирал из реальных имён.
*Зачем:* «задача на Васю» становится связанной с живым человеком: можно открыть чат, пингануть, посчитать «жду от людей» по-настоящему.
*Файлы:* `handler.js`, `analyzer.js` (промпт), схема задачи.

**SMART-003 — Кнопка «Написать» у исполнителя.**
В карточке задачи (Mini App) и в сообщении бота: кнопка открывает чат с исполнителем. Есть username → `https://t.me/username` (в Mini App через `Telegram.WebApp.openTelegramLink`); нет username → `tg://user?id=<tg_id>`.
*Зачем:* один тап от задачи до человека — то самое ощущение «секретаря», а не списка.
*Ограничение:* `tg://user?id=` может не открыться, если пользователь запретил ссылаться на себя — фолбэк: показать имя + «попросить бота напомнить» (см. SMART-012).
*Файлы:* `index.html` (task-detail, BACK-022), `handler.js` (inline-кнопки).

**SMART-004 — Лаконичная фиксация в группах.**
Заменить многословное подтверждение (эмодзи + 3 строки + 2-минутный setTimeout автосохранения) на одну строку: `✓ Вася: подготовить КП — до пт` + две inline-кнопки (✏️ / ✕). В личке можно чуть подробнее, в группе — минимум шума. Убрать setTimeout-механику подтверждения (хрупкая): сохранять сразу, кнопка «✕» отменяет.
*Зачем:* в рабочем чате бот, пишущий простыни, раздражает и его выкидывают. Лаконичность = удержание в группах — а группы это виральный канал (каждый участник чата видит продукт).
*Файлы:* `handler.js:127-136`.

**SMART-005 — Утренняя сводка по чату.**
Подключить готовый `checkBriefings()` (reminders.js:89-120, написан, но не вызывается) + сделать per-group вариант: «Доброе утро! Сегодня: Вася — КП (горит), Аня — макет. Просрочено: 1». Расписание перевести с `setInterval` на Cloudflare Cron Triggers (`[triggers] crons` в wrangler.toml) — надёжнее polling-процесса.
*Зачем:* бот даёт ценность каждый день сам, без запроса — причина не удалять его из чата; заодно готовая инфраструктура для digest-механик виральности (C1).
*Файлы:* `reminders.js`, `index.js`, `wrangler.toml`.

## Блок 2. Умный AI-чат в приложении (реализация «AI-память v1» из ROADMAP)

| # | Задача | Приоритет |
|---|---|---|
| SMART-006 | **Профиль пользователя в контексте AI** | P1 |
| SMART-007 | **Память фактов между сессиями** | P1 |
| SMART-008 | **Действия из чата: не только создать** | P2 |
| SMART-009 | **Один уточняющий вопрос с кнопками** | P2 |
| SMART-010 | **Дедупликация задач** | P3 |
| SMART-014 | **Голосовой разбор нескольких задач и план дня** | P1 (runtime после beta) |

**SMART-006 — Профиль пользователя в контексте AI.**
Сейчас system prompt получает только дату и 12 задач — ни имени, ни статистики. Добавить блок: имя, часовой пояс/город, статистика (всего активных, просрочено, завершено за неделю), люди из задач (топ исполнителей), тариф. Дёшево: данные уже в приложении (`allTasksCache`, профиль).
*Зачем:* «Привет, Юрий. У тебя горит КП для Васи» вместо генерического ответа — мгновенное ощущение личного секретаря. Самый быстрый выигрыш из всего списка.
*Файлы:* `index.html` ~4399-4423 (сборка контекста), ~2819 (SYS_BASE).

**SMART-007 — Память фактов между сессиями (ядро AI-память v1).**
После каждого диалога — фоновый вызов Haiku: «извлеки устойчивые факты о пользователе» (профессия, проекты, постоянные люди, предпочтения: «не ставь ничего на утро субботы»). Хранить в D1 `user_facts (user_id, fact, category, confidence, updated_at)`, топ-15 фактов инжектить в system prompt. Обязателен экран «Что 4 знает обо мне» с удалением фактов — это и 152-ФЗ (прозрачность ПД), и premium trust positioning из ROADMAP.
*Зачем:* чат перестаёт быть «одноразовым»: с каждым днём ответы точнее — главный differentiator и причина удержания.
*Файлы:* `worker.js` (эндпоинты + миграция D1), `index.html`.

**SMART-008 — Действия из чата: не только создать.**
JSON-протокол создания задачи из чата уже есть — расширить набором действий: `complete`, `reschedule`, `edit`, `list`, `remind`. «Перенеси всё некритичное на завтра» → AI возвращает массив действий → приложение показывает превью «Перенесу 3 задачи: … Подтвердить?» → выполняет.
*Зачем:* «умный» = делает, а не советует. Подтверждение обязательно — AI не должен молча менять данные.
*Файлы:* `index.html` (парсер ответа AI + обработчики), SYS_BASE.

**SMART-009 — Один уточняющий вопрос с кнопками.**
Если в задаче нет дедлайна или исполнитель неоднозначен — не длинный текст, а короткий вопрос с кнопками-вариантами: «Когда? [Сегодня] [Завтра] [Пятница] [Без срока]». Максимум один вопрос, дальше сохраняем как есть.
*Зачем:* лаконичность + полнота данных без допроса. Кнопки быстрее набора текста — мобильный паттерн.
*Файлы:* `index.html`, `handler.js` (в боте тот же паттерн).

**SMART-013 — «Задача — это промпт»: AI-декомпозиция на этапы (идея Юрия, 2026-07-04).**
Кнопка «Разбить на этапы» в карточке: AI разворачивает задачу в чек-лист шагов (используя чек-лист из BACK-022); в чате задачи (BACK-032) можно углубиться в любой шаг («распиши подробнее», «предложи варианты»). Стартовый набор шаблонов декомпозиции: запуск продукта, отчёт, событие, ремонт/переезд. Механика чек-листа уже есть — нужен AI-генератор наполнения.
ТЗ: `docs/tasks/SMART-013-ai-task-decomposition.md`. MVP не вводит отдельные подзадачи: AI показывает preview, пользователь подтверждает, этапы сохраняются как обычные пункты чек-листа.
*Зачем:* переворачивает категорию — не «записал и забыл», а «записал — получил план». Главный кандидат на wow-момент онбординга (C5/VIRAL-005) и основа вирального ролика 2Б.
*Файлы:* `index.html` (task-detail), SYS_BASE, позже worker.

**SMART-014 — Голосовой разбор нескольких задач и план дня.**
Одна голосовая фраза с несколькими делами превращается в редактируемый preview из 2–12 самостоятельных задач. 4 предлагает один изменяемый главный фокус и показывает связанные дела рядом, но не склеивает их. Сроки, длительность, поездки и свободные окна не выдумываются. Сохранение выполняется только после точного CTA `Добавить N задач`; нужен идемпотентный batch path. Эталонный сценарий одновременно уточняет политику `SMART-009`: вопрос задаётся только при неопределённости, которая существенно меняет данные или смысл плана.
ТЗ: `docs/tasks/SMART-014-voice-multi-task-day-plan.md`.
*Зачем:* голос становится не заменой клавиатуры, а сильной Premium-функцией личного штаба дня.
*Очередность:* стратегически P1; runtime после `BETA-001` и стабильного `BACK-021`, не blocker текущей beta.
*Файлы:* `index.html` (`sendVoiceMessage`, AI chat/task creation), Worker `/anthropic`, новый batch save path, analytics.

**SMART-010 — Дедупликация задач.**
При создании — быстрый чек на похожесть с активными (нормализованный текст + AI при сомнении): «Похоже на "созвон с Васей по КП" от вторника. Объединить или создать новую?»
*Зачем:* дубли — главный мусор в планерах, из-за них дашборд «врёт» и доверие падает.

## Блок 3. Проактивность

| # | Задача | Приоритет |
|---|---|---|
| SMART-011 | **Умный пинг «Жду от людей»** | P2 |
| SMART-012 | **Адаптивное время напоминаний** | P3 (Горизонт 1–2, AI-память v2) |

**SMART-011 — Умный пинг «Жду от людей».**
Для задач с `assigneeTgId` (после SMART-002): кнопка «Напомнить Васе» — бот шлёт исполнителю вежливое напоминание от своего имени (или готовит черновик сообщения пользователю для отправки самому). Утром: «Ты ждёшь от людей 3 задачи, 1 просрочена. Напомнить?»
*Зачем:* закрывает самую болезненную рутину делегирования — «неудобно пинговать». Секретарь делает это за тебя.

**SMART-012 — Адаптивное время напоминаний.**
Учиться на поведении: когда пользователь реально закрывает задачи и читает сообщения — туда и сдвигать брифинг/напоминания вместо фиксированных интервалов. Часть AI-память v2.
