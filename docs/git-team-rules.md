# Git — правила команды

Документ для Алексея, Юрия и агентов. Цель — держать проект живым без случайных push/merge в live и без лишнего ограничения опытных участников.

## Главный принцип

`main` связан с GitHub Pages. Всё, что попадает в `main`, может быстро стать live-версией продукта.

Поэтому:
- Codex, Claude и другие агенты не делают push, merge и рискованный `pull --rebase` молча.
- Локальный commit можно делать по явной просьбе.
- GitHub Desktop — удобный вариант для Алексея и слабых пользователей Git, но не обязательное правило для Юрия.
- Непонятные ситуации не решаются наугад.

## Ветки

| Тип работы | Ветка |
| --- | --- |
| Багфикс | `fix/короткое-описание` |
| Новая функция | `feat/короткое-описание` |
| Документы / процесс | `docs/короткое-описание` |
| Срочный P0 hotfix | Только после отдельного решения Алексея / Юрия |

## Старт работы

1. Проверить текущую ветку.
2. Проверить изменённые файлы.
3. Синхронизироваться с remote только если понятно, что текущие изменения не потеряются.
4. Если есть изменённые файлы — сначала понять, кто и зачем их изменил.

Если непонятно, что происходит, не нажимать дальше. Сначала спросить Codex / Юрия.

Для Алексея безопасный вариант — делать эти шаги через GitHub Desktop: выбрать ветку, нажать `Fetch origin`, затем `Pull` только если Desktop не показывает конфликтов и непонятных изменений.

## Перед commit

Проверить:
- задача понятна;
- изменённые файлы относятся к задаче;
- нет случайных временных файлов;
- если менялся код, есть ручная проверка или тест;
- если задача значимая, обновлены `DEVELOPMENT_LOG.md` и `shared/WORK_LOG.md`.

Название commit писать по правилу:

```text
type(scope): что изменилось
```

Примеры:

```text
docs(process): define branch review protocol
fix(auth): prevent empty reset screen
feat(tasks): show relative due dates
```

## Push

Push делать только после отдельного согласования. По умолчанию push идёт в рабочую ветку, не напрямую в `main`, если это не согласованный hotfix.

Перед push:
1. Проверить ветку.
2. Проверить список файлов.
3. Убедиться, что проверка прошла.
4. Получить согласование, если push может затронуть live или чужую работу.

## Завершение задачи и синхронизация

По завершению задачи агент помогает максимально автоматизировать рутину:

1. Проверить изменения.
2. Сделать commit с понятным заголовком.
3. Сделать push в рабочую ветку.
4. Обновить `pm/team-sync.md`.
5. Сообщить ветку, commit/PR, проверку и следующий шаг.

Это правило не означает автоматический merge в `main`. `main` связан с GitHub Pages, поэтому merge выполняется отдельно и только после подтверждения Алексея или Юрия.

Если commit сделан, но push не прошёл, задача не считается переданной команде. Нужно написать причину и дать простую ручную инструкцию.

## Team Sync

Короткие статусы между Алексеем, Юрием, Codex и Claude ведутся через `pm/team-sync.md`.

Ключевые фразы:
- Юрий спрашивает Claude: `Что там у Лехи?`
- Алексей спрашивает Codex: `Что там у Юры?`
- После завершения задачи: `Закрой задачу и синхронизируй`

Подробное правило: `docs/team-sync-protocol.md`.

## Merge в main

Merge разрешён только если:
1. Задача завершена.
2. Понятно, что изменилось.
3. Проверка выполнена.
4. Нет незавершённых файлов.
5. Алексей или Юрий явно согласовали merge.

Предварительный рабочий вариант правила: разбирать готовые ветки 1 раз в неделю + срочно для P0/P1. Это ещё не финальное правило, пока Алексей не утвердит его после мнения Юрия / Claude.

## Исключения

Исключения возможны, но только после согласования.

Перед исключением нужно явно написать:
- какое правило нарушаем;
- зачем это нужно;
- что может пойти не так;
- кто подтвердил решение.

Пример:

```text
Исключение: делаем hotfix прямо в main.
Причина: P0, live-версия не открывается.
Риск: push сразу повлияет на пользователей.
Подтвердил: Алексей.
```

## Когда обязательно звать человека

- Git показывает conflict.
- Есть незакоммиченные изменения непонятного происхождения.
- Нужно push/merge в `main`.
- Нужно force push.
- Решение влияет на стратегию, цену, юридические обещания, безопасность или данные пользователей.
