# UI-архитектура

Эти правила защищают текущий Mini App, пока проект постепенно выходит из большого `index.html` к модульному фронтенду.

## Решение

Новый UI-код пишем по разделению:

- HTML — структура и семантика.
- LESS — визуальные стили.
- Frontend JS — поведение.
- BEM-подход в классах показывает, какому блоку принадлежит элемент.

## Где действует

- `index.html`
- `vk.html`
- файлы в `styles/`
- будущие frontend-модули, вынесенные из Mini App

Правило не относится к Worker, боту, SQL-миграциям, deploy-конфигам и PM-документам.

## Обязательно для нового UI-кода

1. Новые визуальные стили добавлять в `styles/**/*.less`, затем запускать `npm run build:css`.
2. Новые UI-классы называть в BEM-подходе: `block`, `block__element`, `block--modifier`.
3. Не добавлять новые `style=""` в HTML. Существующие inline-стили считаются legacy-долгом.
4. Не добавлять новые `onclick`, `oninput`, `onchange` и похожие inline-обработчики. Использовать `addEventListener()` или делегирование событий.
5. Не добавлять новые inline `<style>` и inline `<script>` блоки в `index.html`.
6. Сохранять подключение `styles.min.css` как production-стилей.

## Старый код

Старый inline-код не переписываем одним большим проходом. Правило: не увеличивать долг. Когда экран трогаем по продуктовой задаче, близкие стили и обработчики переносим в LESS и JS в рамках этой же ограниченной правки.

Технический guard — `scripts/check-ui-architecture.sh`. Он допускает текущий legacy-baseline, но падает, если inline-долг растёт.

## Исключения

Исключение допустимо только при выполнении всех условий:

- это нужно для маленького срочного фикса;
- причина записана в задаче или commit notes;
- создан follow-up на очистку или baseline guard-а явно пересмотрен.
