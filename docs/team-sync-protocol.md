# Team Sync Protocol

Короткий протокол для Алексея, Юрия, Codex и Claude. Цель - чтобы команда видела последние задачи и изменения через GitHub и один диспетчерский файл, без ручного пересказа в чатах.

## Главный принцип

`pm/team-sync.md` - короткий центр синхронизации команды.

Он не заменяет:
- `shared/ROADMAP.md` - стратегию и направления продукта;
- `pm/backlog.md` - список задач;
- `pm/bugs.md` - баги и QA-входящие;
- `shared/WORK_LOG.md` и `DEVELOPMENT_LOG.md` - журналы выполненной работы.

`pm/team-sync.md` показывает только то, что нужно быстро понять между людьми и агентами: кто что передал, кто что сделал, что заблокировано и какой следующий шаг.

## Ключевые фразы

### Для Юрия

```text
Что там у Лехи?
```

Claude Юры должен сначала прочитать этот протокол и `pm/team-sync.md`, затем дать короткий отчет: что Алексей передал Юре, что важно сейчас, что заблокировано и что нужно сделать дальше.

```text
Закрой задачу и синхронизируй.
```

Claude Юры должен проверить изменения, подготовить commit, push в рабочую ветку и обновить `pm/team-sync.md`. Если push не получился, нужно прямо написать причину и дать Юре простую ручную инструкцию.

```text
Что мне делать дальше?
```

Claude Юры должен смотреть `pm/team-sync.md`, `pm/backlog.md`, `pm/bugs.md`, `shared/ROADMAP.md` и предложить ближайший практический шаг без длинной теории.

### Для Алексея

```text
Что там у Юры?
```

Codex должен проверить `pm/team-sync.md`, последние коммиты, `pm/backlog.md`, `pm/bugs.md`, `shared/ROADMAP.md` и `shared/WORK_LOG.md`, затем дать отчет: что Юра сделал, что запушено, что требует проверки, что заблокировано и какие решения нужны от Алексея.

```text
Дай статус проекта.
```

Codex должен собрать короткий PM-отчет по roadmap, backlog, багам, блокерам и ближайшему следующему шагу.

## Что агент читает перед отчетом

При фразах `Что там у Лехи?`, `Что там у Юры?`, `Дай статус проекта` агент читает:

1. `pm/team-sync.md`
2. `pm/backlog.md`
3. `pm/bugs.md`
4. `shared/ROADMAP.md`
5. `shared/WORK_LOG.md`
6. `DEVELOPMENT_LOG.md`, если нужен технический контекст
7. `git status --short --branch`
8. последние коммиты текущей ветки и `origin/main`, если нужно понять свежесть изменений

## Когда задача считается завершенной

Задача считается завершенной только если:

1. изменения относятся к одной понятной задаче;
2. выполнена проверка или ручной smoke, подходящий масштабу задачи;
3. сделан commit с понятным заголовком по `shared/COMMIT_CONVENTION.md`;
4. сделан push в рабочую ветку;
5. обновлен `pm/team-sync.md`;
6. в отчете указаны ветка, commit, push/PR и следующий шаг.

Если commit сделан, но push не прошел, задача не считается переданной команде. Агент должен объяснить причину и дать простую инструкцию, что сделать вручную.

## Статусы

Используем только короткий список:

- `Передано`
- `В работе`
- `Сделано локально`
- `Запушено`
- `На проверке`
- `Заблокировано`
- `Готово к main`
- `В main`

Если статуса не хватает, лучше написать короткое пояснение в строке `Блокер` или `Следующий шаг`, а не плодить новые статусы.

## Git-границы

- Commit и push делаем по завершению задачи в рабочую ветку.
- Push напрямую в `main` не делаем без отдельного подтверждения.
- Merge в `main` делается отдельно, только после подтверждения Алексея или Юры.
- Если Git показывает конфликт, непонятную ветку или чужие незакоммиченные изменения, агент останавливается и выносит решение человеку.

## Инструкция Юре без погружения в Git

Юра, тебе не нужно разбираться в тонкостях Git, чтобы процесс работал. Основная идея простая: ты говоришь Claude короткую фразу, а Claude сам смотрит нужные файлы и подсказывает следующий шаг.

Команды для Claude:

```text
Что там у Лехи?
```

Понять, что Алексей передал тебе в работу.

```text
Что мне делать дальше?
```

Получить ближайшую задачу и контекст.

```text
Закрой задачу и синхронизируй.
```

Попросить Claude проверить изменения, сделать commit, push и обновить `pm/team-sync.md`.

```text
Сделай короткий отчет Алексею.
```

Получить готовый текст: что сделано, где ветка/commit, что проверить и что дальше.

Первым делом после появления этого протокола в GitHub:

1. Обнови локальный проект через GitHub Desktop: `Fetch origin`, затем `Pull`, если Desktop не показывает конфликтов.
2. Напиши Claude:

```text
Прими правила синхронизации команды: прочитай CLAUDE.md, docs/team-sync-protocol.md и pm/team-sync.md. Дальше при фразе "Что там у Лехи?" работай по этим правилам.
```

3. После ответа Claude используй короткие фразы выше.

Юмор по делу: Git пусть таскает Claude, а ты таскай продукт вперед. Если Git начинает умничать - не спорь с ним, проси Claude объяснить одним абзацем, что нажать.

## Формат короткого отчета

```text
Статус:
...

Что сделал Алексей / Юра:
...

Ветка / commit / PR:
...

Что проверить:
...

Блокеры:
...

Следующий шаг:
...
```
