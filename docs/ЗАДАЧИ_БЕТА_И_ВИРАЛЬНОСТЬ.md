## 1. Диагноз: что тормозит разработку

**Монолит.** index.html — ~6000 строк (CSS + JS + HTML в одном файле). Каждая правка рискованна, отсюда и правило «ШАГ 0» с проверкой кириллицы — это симптом, а не решение. vk.html дублирует ~70% кода: каждую фичу делаем дважды, баги фиксим дважды.

**Нет staging.** Worker деплоится сразу в prod (`wrangler deploy`), Pages — из main сразу в live. wrangler.staging.toml есть, но staging D1/KV не созданы. Любая ошибка агента сразу видна пользователям.

**Нет автотестов и CI.** Весь QA ручной (Алексей на телефонах). Регрессии типа BUG-06-25-001 ловятся уже в live.

**Две папки-копии** (Documents\4 актуальная, Desktop\4\Версия — копия) — риск правок не туда и расхождения версий.

## 2. Чистая vs грязная версия: вердикт

**Идея правильная, реализация через две копии кода — нет.** Держать два независимых кода (чистый для людей, грязный для доработок) для монолита в 6000 строк — это ручной перенос правок между копиями, двойные конфликты и гарантированное расхождение (это уже происходит с Documents/Desktop).

**Как правильно (тот же эффект, без двойной работы):**

| Слой | «Чистая» | «Грязная» |
|---|---|---|
| Код | ветка `main` | ветки `feat/*`, `fix/*`, ветка `dev` |
| Mini App | GitHub Pages (main) | второй Pages-сайт / Cloudflare Pages preview из `dev` |
| Worker | prod deploy | `wrangler deploy --env staging` + свои D1/KV |
| Бот | @Denzel89bot | второй тестовый бот на тот же staging-worker |

Пользователи всегда видят main. Агенты ломают что угодно в dev/staging. Merge в main — только после QA-чеклиста. Это и есть «чистовая» версия, но с одним источником правды.

**Дополнительно:** для рискованных фич — feature-флаги (`?debug=1` или флаг в KV), чтобы недоделанное жило в main выключенным.

## 3. Виральность AI-планера: анализ

Что уже есть: AI-планер-дашборд (BACK-025, in progress), glass UI — «инстаграмный» вид, которым захочется делиться. Контент-стратегия (VK Клипы + AI-персонаж) спланирована, но **в продукте нет ни одной виральной механики** — ни шеринга, ни рефералок, ни streaks (проверено по коду).

Ключевой принцип: виральность планера = **красивый артефакт дня, которым хочется поделиться** (план утром, итоги вечером) + **выгода за приглашение**. Персонаж AI-блогера должен жить и внутри приложения (модель Duolingo), иначе контент в VK Клипах не будет узнаваться в продукте.

Важно: рефералки бессмысленны, пока дубли аккаунтов (BACK-026) не решены — приглашённый по ссылке из VK, зашедший потом через TG, потеряется.

### D. Новые направления (добавлено 2026-07-04)

**AI-память — упор на логику ответов ИИ.** Ассистент должен накапливать опыт пользователя и использовать его в ответах. Поэтапно, чтобы не строить всё сразу:

| # | Задача | Горизонт | Зачем / польза |
|---|---|---|---|
| D1 | **AI-память v1: профиль опыта** — таблица в D1 (факты, предпочтения, паттерны из задач и чата), инжект в контекст AI-сводки и чата | 0.5–0.8 | Ответы AI перестают быть «одинаковыми для всех» — главный differentiator от Todoist/Notion |
| D2 | **AI-память v2: обучение на поведении** — что завершает/переносит, когда продуктивен → персональный порядок «Начать с» | 1 | Планер становится точнее с каждым днём использования — причина не уходить |
| D3 | **AI-память v3: предиктивность** — авто-шаблоны («каждый понедельник standup — ставить автоматически?»), предиктивные напоминания | 2 | Столп 3 (автоматизация); уже был в ROADMAP, теперь связан с v1–v2 |

Важно: память = ПД. Прозрачность (экран «что 4 знает обо мне» с возможностью удалить) — и требование 152-ФЗ, и часть premium trust positioning.
