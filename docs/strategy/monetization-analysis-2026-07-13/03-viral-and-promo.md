# 3. Виральные функции и промо

## 3.1. Принцип

Виральность 4 должна быть следствием полученной ценности, а не отдельной игровой системой.

Рабочая петля:

`пользователь получил полезный план → возник естественный повод показать результат → друг открыл персональную ссылку → создал свой первый план → бонус подтверждён → новый пользователь повторяет цикл`

Если share происходит до результата, это реклама без доверия. Если бонус выдаётся до активации, это источник накрутки. Если приглашённый не получает план за минуту, referral traffic теряется.

## 3.3. `ONBOARD-001` важнее referral и share-card

Текущий onboarding показывает четыре информационных слайда и переводит на login. Пользователь:

- не добавляет свои дела;
- не получает персональный результат;
- не видит разницу между 4 и обычным списком;
- не создаёт событие активации, на котором можно строить paywall/referral.

Минимальный first-value flow:

1. Регистрация/вход.
2. Экран «Добавь 3 дела».
3. Выбор «сказать голосом» или «ввести текстом»; голос не единственный путь.
4. Прогресс `1/3 → 2/3 → 3/3`.
5. Открывается настоящий Home с реальными задачами.
6. Подсветка: «Вот твой план. Начни с этого».
7. События `onboarding_started`, `task_n_created`, `tasks_3_created`, `first_plan_viewed`, `onboarding_completed`.

`VIRAL-005` не вести как отдельную разработку. Это алиас growth-эффекта `ONBOARD-001`.

## 3.4. Safe-MVP реферальной программы

### Что уже есть

- referral code и counter в профиле;
- capture параметров `ref/referral/invite`;
- pending code между входом и регистрацией;
- передача ref в email/TG/VK/OAuth auth-flow;
- backend начисляет 30 дней обоим, запрещает self-referral и повторный `referredBy`;
- inviter получает `referralCount`.

### Чего не хватает

- реферальные frontend/backend коммиты не в `origin/main`;
- production app и VK build не содержат flow;
- нет smoke `inviter → invited → activation → reward`;
- нет `source/campaign` attribution;
- нет событий `referral_link_copied`, `referral_opened`, `referral_signup`, `referral_activated`, `referral_rewarded`, `referral_paid`;
- нет лимита, velocity/device heuristics или manual review;
- бонус выдаётся сразу за новый аккаунт, а не за ценность;
- 30 дней обоим вместе с 30-дневным trial увеличивают бесплатный cost-to-serve и отодвигают оплату.

### Рекомендация

Для beta сохранить текущую механику как QA-flow, но до public launch изменить момент reward:

**Вариант A — рекомендуемый для старта:** reward после `first_plan_viewed` и возврата D1.

**Вариант B — более монетизационный:** бонус приглашённому сразу, пригласившему — после первой оплаты приглашённого.
**Вариант C — текущий:** обоим за регистрацию; быстрее кодом, но самый уязвимый и хуже для выручки.

Нужен один явный выбор команды. Не менять условия молча.

## 3.5. VIRAL-001: минимальная карточка, а не дизайн-проект

Первая версия должна быть детерминированной и полезной:

- заголовок `Мой план на сегодня` или `Итоги дня`;
- 3 безопасных показателя: задач запланировано/выполнено, главный фокус, серия только если она реальна;
- настройка приватности: по умолчанию не показывать текст задач, имена людей, проекты и дедлайны;
- бренд 4;
- QR/deep link с `source=share_card`, `campaign=plan_day`, referral code;
- экспорт PNG + системный Web Share;
- VK Story adapter отдельным шагом;
- Telegram — share link/photo через доступную платформенную механику, без обещания API, которого продукт не вызывает.

MVP можно сделать без AI-генерации. LLM-копирайтинг добавит стоимость и непредсказуемость, но не докажет сам share-loop.

## 3.6. Что реально есть для промо

### В каноническом app repo

- семь сценариев роликов на 15–30 секунд;
- порядок публикации и ритм два ролика в неделю;
- SMART-013 mockup и notification wireframe как продуктовые references;
- ни одного MP4/MOV/WebM;
- нет character image, voice sample, social templates, banner exports или готовой conversion page.

### В старой desktop-копии

В `<legacy-desktop-copy>` найдены:

- `roadmap_ai_bloger.md`;
- `4-ai-blog/VK_CONTENT_PLAN.md` с 10 постами;
- `4-ai-blog/index.html`, три SEO-статьи, sitemap, robots и deploy-инструкция;
- дубликат сценариев роликов.

Полезный контент нельзя терять, но публиковать пакет как есть нельзя:

- CTA ведут на `t.me/your_bot`;
- объявлен `og-image.png`, которого нет;
- есть ссылки на отсутствующую в blog package `privacy.html`;
- позиционирование Telegram-only противоречит текущему «омниканальному штабу дня»;
- встречаются утверждения «в 3 раза быстрее» и «теряется 80% задач» без источников;
- testimonial «Алексей, Москва» нельзя использовать без подтверждения и согласия;
- сценарий говорит «Бесплатно», хотя продукт предлагает ограниченный Premium trial.

### Что сделать с этими материалами

Не переписывать с нуля. После выбора канонического promo-source:

1. Скопировать материалы в versioned `docs/marketing/` отдельной задачей.
2. Удалить placeholders и недоказанные claims.
3. Обновить позиционирование и CTA.
4. Подключить каноническую privacy page.
5. Добавить tracking links.
6. Публиковать только те feature claims, которые видны на release candidate.

## 3.7. Первые два промо-эксперимента

### Ролик A — голос → две задачи

Почему: реальный, короткий, легко снять screen recording. До записи нужен живой sample текущей latency: telemetry показала искусственную UI-паузу около 3,6 секунды после распознавания. Нельзя обещать «5 секунд», если видео это опровергает.

Структура:

- 0–2 сек: «Пока ты открываешь заметки, мысль уже потерялась»;
- 2–10 сек: реальная диктовка;
- 10–18 сек: две реальные задачи с датами;
- CTA: «Собери первый план в 4».

### Ролик B — задача превращается в шаги

Почему: `SMART-013` прошёл staging smoke и лучше всего доказывает «не ещё один todo».

Структура:

- 0–2 сек: «Записал “запустить продукт”. И что дальше?»;
- 2–12 сек: реальная декомпозиция;
- 12–20 сек: пользователь подтверждает список шагов;
- CTA: «Проверь на своей задаче».

Не показывать chat-depth `BACK-032`, пока этот кусок не прошёл production smoke.

## 3.8. Promo funnel и события

Каждая ссылка должна содержать минимум:

- `source`: `vk_clip`, `tg_channel`, `warm_invite`, `share_card`;
- `campaign`: конкретный ролик/публикация;
- `ref`: если это персональная ссылка;
- platform entry: правильный TG/VK/web путь.

Воронка:

`clip_view (platform) → profile_click → app_open → signup_completed → first_task → first_plan → D1_return → paywall_view → checkout_started → payment_success`

Без этой цепочки просмотры и досмотры не говорят, приблизился ли проект к выручке.

## 3.9. Приоритеты

### Сейчас

1. `ONBOARD-001` + реальные activation events.
2. Referral smoke и reward-after-activation decision.
3. Канонизация promo-source.
4. Два screen-first ролика.
5. Одна простая conversion entry с реальной ссылкой, privacy и tracking.

### После beta

1. `VIRAL-001`.
2. Streak как поле внутри карточки, если пользователи реально возвращаются.
3. `VIRAL-006` на детерминированной статистике.
4. Минимальный character bible и статичный avatar.
5. SEO-блог после подтверждения message-market fit.

### Не делать сейчас

- полноценного анимированного AI-блогера;
- отдельную points/badges economy;
- LLM/cron weekly summary;
- paid traffic;
- массовый SEO;
- рекламу функций, не вошедших в release candidate;
- публикацию старого desktop-blog без ревизии.
